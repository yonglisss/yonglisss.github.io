Technical Guide

02. 快速开始：先把 DeerFlow 跑起来

用最短路径完成 DeerFlow 2.0 的本地配置、环境检查、Docker / 本地启动和第一个任务验证。

目标

这一篇只做一件事：先把 DeerFlow 跑起来。

不要一开始就研究全部源码。Agent 框架只有跑起来之后，配置、工具、沙盒、子 Agent 这些概念才会变得具体。

完成后，你应该得到一个可访问的 DeerFlow Web 页面，并能发起第一个任务。

环境要求

官方 README 里给出的基础要求是：

  • Python 3.12+
  • Node.js 22+
  • Docker（推荐，尤其是要用 sandbox 时）
  • pnpm、uv、nginx 等本地开发依赖

如果只是先体验，建议优先用 Docker 路线。它更接近官方推荐，也能少踩本地依赖版本的坑。

资源上可以按这个起步：

场景起步配置更舒服的配置
本地体验 / make dev4 vCPU / 8 GB 内存 / 20 GB 可用空间8 vCPU / 16 GB 内存
Docker 开发 / make docker-start4 vCPU / 8 GB 内存 / 25 GB 可用空间8 vCPU / 16 GB 内存
长期运行服务 / make up8 vCPU / 16 GB 内存 / 40 GB 可用空间16 vCPU / 32 GB 内存

注意：如果你还要在本机跑本地大模型，需要单独给模型服务预留资源。

1. 克隆项目

git clone https://github.com/bytedance/deer-flow.git
cd deer-flow

后续命令默认都在项目根目录执行。

2. 生成配置

官方现在更推荐先走交互式配置：

make setup

这个向导会帮你选择：

  • 使用哪个 LLM provider
  • 是否配置搜索能力
  • 是否启用 sandbox
  • 是否允许 bash / 文件写入等执行能力

它会生成最小可用的 config.yaml，并把密钥写入 .env。

如果你不想走向导，也可以用完整模板：

make config

这会基于 config.example.yaml 生成本地配置。适合你已经知道要怎么改模型、搜索、sandbox 的情况。

3. 配置模型 provider

DeerFlow 至少需要一个可用模型。

最常见的是 OpenAI 兼容配置：

models:
  - name: gpt-4o
    display_name: GPT-4o
    use: langchain_openai:ChatOpenAI
    model: gpt-4o
    api_key: $OPENAI_API_KEY

如果你使用 OpenRouter、火山方舟、内部网关这类 OpenAI 兼容服务，通常也是 langchain_openai:ChatOpenAI，再加 base_url：

models:
  - name: openrouter-gemini-2.5-flash
    display_name: Gemini 2.5 Flash (OpenRouter)
    use: langchain_openai:ChatOpenAI
    model: google/gemini-2.5-flash-preview
    api_key: $OPENROUTER_API_KEY
    base_url: https://openrouter.ai/api/v1

API key 推荐放在 .env：

OPENAI_API_KEY=your-o...-key
OPENROUTER_API_KEY=your-or...-key

不要把真实 key 直接提交进仓库。

4. 配置搜索能力

搜索不是所有任务都必需，但对研究类任务很关键。

make setup 会询问你是否配置 web search provider。你也可以之后手动补充对应环境变量，例如：

TAVILY_API_KEY=your-t...-key
INFOQUEST_API_KEY=your-i...-key

怎么判断要不要先配搜索？

  • 只测试对话、文件处理、代码执行：可以先不配。
  • 要做资料研究、网页检索、趋势分析：建议配。
  • 要跑官方演示里的研究型任务：建议配。

先跑通系统时，不要被搜索配置卡太久。模型能用，是第一优先级。

5. 检查配置

执行：

make doctor

它会检查当前配置是否可用，并给出可执行的修复建议。

这一步不要跳过。很多“启动失败”本质上是：

  • config.yaml 不在项目根目录
  • .env 没有对应 API key
  • provider 名称或模型名写错
  • Docker 没启动
  • sandbox 镜像还没拉下来

如果你从别的目录启动 DeerFlow，可以显式指定项目根目录：

export DEER_FLOW_PROJECT_ROOT=/path/to/deer-flow

也可以直接指定配置文件：

export DEER_FLOW_CONFIG_PATH=/path/to/deer-flow/config.yaml

6. 启动方式一：Docker 开发模式

推荐先用 Docker。

首次执行：

make docker-init

它会拉取 sandbox 相关镜像。镜像比较大，第一次可能需要等一会儿。

然后启动：

make docker-start

启动后访问：

http://localhost:2026

Docker 开发模式适合：

  • 想少处理本地依赖
  • 想使用 sandbox
  • 想接近真实部署环境
  • 不介意镜像构建和容器占用资源

如果你在 Linux 上遇到 Docker socket 权限问题，通常是当前用户没有访问 Docker daemon 的权限。需要把用户加入 docker 组并重新登录。

7. 启动方式二：本地开发模式

如果你要直接在本机跑服务，先检查依赖：

make check

安装依赖：

make install

如果使用 Docker / Container sandbox，建议先拉镜像：

make setup-sandbox

启动：

make dev

访问：

http://localhost:2026

本地开发模式适合：

  • 你要改源码
  • 想快速看日志和热更新
  • 已经能稳定处理 Python、Node、pnpm、uv、nginx 等依赖

Windows 用户建议用 Git Bash。官方说明里提到，基于 bash 的服务脚本不支持直接在原生 cmd.exe 或 PowerShell 中执行。

8. 跑通第一个任务

打开页面后，先发一个低风险任务：

请用 5 条要点介绍 DeerFlow 2.0 的核心能力，不需要联网。

如果模型配置正常，你应该能看到回复。

然后再试一个需要执行能力的任务：

创建一个简单的 Markdown 大纲，主题是“DeerFlow 入门学习路线”，并保存成文件。

如果你开启了 sandbox 或文件写入能力，这类任务可以帮助你验证：

  • Agent 是否能规划步骤
  • 文件系统能力是否可用
  • 执行环境是否按预期工作

最后再试研究型任务：

搜索 DeerFlow 2.0 的官方资料，整理它和普通聊天机器人的区别。

如果这一步失败，通常是搜索 provider 没配好；不影响基础系统已经跑通。

常见卡点

config.yaml 找不到

确认它在项目根目录：

deer-flow/config.yaml

或者设置：

export DEER_FLOW_PROJECT_ROOT=/path/to/deer-flow

API key 明明写了但没生效

优先检查 .env 中变量名和 config.yaml 里的引用是否一致。

例如配置里写的是：

api_key: $OPENROUTER_API_KEY

.env 里就必须有：

OPENROUTER_API_KEY=***

Docker 模式启动慢

第一次拉 sandbox 镜像会慢。可以先执行：

make setup-sandbox

这样能看到更明确的下载进度。

模型可用，但联网任务失败

这通常不是模型问题，而是搜索 provider 没配置。先用不联网任务验证主链路，再回头补搜索配置。

这一篇的完成标准

你不需要一次性把所有高级能力都配完。

只要满足下面三点，就可以进入下一篇：

  • make doctor 没有阻塞性错误。
  • Docker 或本地开发模式能启动，并能访问 http://localhost:2026。
  • 至少跑通一个不联网的简单任务。

下一篇会开始拆 config.yaml。DeerFlow 的很多能力边界，其实都藏在配置里。
