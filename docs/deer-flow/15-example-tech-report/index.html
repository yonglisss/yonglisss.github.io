15. 实战二：用 DeerFlow 生成一份技术报告

用 DeerFlow 把一个技术主题整理成 Markdown 报告，重点控制结构、事实来源、风格和最终落盘。

实战目标

这一篇做一个更接近日常工作的任务：输入一个技术主题，让 DeerFlow 生成一份可保存、可继续编辑的技术报告。

和上一篇深度研究不同，这里更关注报告交付质量：结构、事实、表达、文件落盘。

示例任务

可以选择一个具体主题：

请生成一份“DeerFlow 2.0 本地部署注意事项”的技术报告。

不要让主题太大。太大的主题容易变成泛泛而谈。

推荐提示词

请基于 DeerFlow 官方 README、Install.md、config.example.yaml 和 backend/docs/SETUP.md，生成一份“DeerFlow 2.0 本地部署注意事项”的 Markdown 技术报告。

要求：
1. 先读取官方资料，不要基于记忆写。
2. 报告结构包括：环境要求、配置文件、模型配置、搜索配置、Docker 启动、本地开发、常见错误、验证清单。
3. 命令必须使用代码块。
4. 不要写营销表达。
5. 最后保存为 deerflow-deployment-notes.md。

这个提示词把来源、结构、风格和文件产物都约束住了。

控制报告结构

技术报告最怕两种问题：太散，看起来像聊天总结；太满，把所有资料原样堆进去。

建议固定结构：背景、前置条件、操作步骤、验证方式、常见问题、下一步。

结构稳定，后续才容易比较质量。

控制事实来源

让 Agent 写技术报告时，要明确哪些资料是主来源，哪些资料只能作为辅助，不确定的地方要标注，没读取到原文就不要写确定结论。

这比“写得专业一点”更重要。

控制风格

技术报告不需要夸张。

建议要求：短句、少形容词、多步骤、多验证、少口号，不写“革命性”“颠覆式”这类词。

真正有用的技术报告，是读者照着能做事。

验证文件产物

如果要求保存文件，不要只看最终回复。

要检查文件是否存在、文件名是否正确、Markdown 格式是否正常、代码块是否闭合、命令是否来自官方资料、是否可以继续提交或发布。

人工介入点

即使 Agent 生成了报告，人也应该检查版本是否和官方最新一致、命令是否真实可运行、API key 示例是否泄露真实值、安全建议是否足够、是否有过时信息。

Agent 适合做初稿和结构化整理，不适合替代最终技术审查。

常见失败点

  • 报告只写概念，没有命令；
  • 命令来自旧版本文章；
  • 没有验证步骤；
  • 代码块格式错误；
  • 文件路径不明确；
  • 内容太像营销稿。

复盘

这个实战的目标不是让 DeerFlow 写一篇好看的文章，而是产出一个能继续使用的工程文档。

如果报告能被保存、被检查、被修改、被发布，它就从聊天回答变成了工作产物。
