Technical Guide

10. Memory：长期记忆如何参与任务

理解长期记忆：哪些信息值得记、哪些不能记、如何避免污染，以及它和上下文、skill 的区别。

Memory 不是聊天记录仓库

长期记忆听起来很诱人：Agent 记住用户、项目、偏好，下次自动接上。

但 Memory 不是越多越好。错误记忆、过期记忆、临时进度一旦被长期保存，会让 Agent 下次继续犯错。

所以学习 Memory，第一件事不是看怎么存，而是看什么值得存。

Memory 解决什么问题

模型上下文有限。如果所有历史都靠当前对话传递，用户需要反复说明偏好，项目固定路径每次都要重复，常用流程容易忘。

Memory 的作用，是保存长期稳定的信息，让 Agent 下次能直接使用。

适合进入 Memory 的信息

适合保存的信息通常长期有效、多次任务会用到、已经被用户确认、不容易从当前环境直接查到、能减少未来沟通成本。

例如：用户偏好中文短回复、某项目常用路径、代码提交作者邮箱、内容写作不营销、知识库默认目录。

不适合进入 Memory 的信息

某次任务进度、临时 bug、提交 hash、PR 编号、一周后会过期的数据、未验证猜测，都不适合。

这些更适合保存在当前会话、项目文档、issue、commit 或任务系统里。

Memory 和当前上下文的区别

当前上下文是这次任务正在使用的信息。Memory 是长期背景。

Memory 给方向，工具给事实。如果 Memory 说项目路径是 A，但当前文件系统里路径变成了 B，应该相信当前工具检查结果。

Memory 和 Skill 的区别

Memory 保存事实，Skill 保存流程。

“用户偏好默认中文沟通”是 Memory。
“写教程时先查官方文档，再运行构建验证”是 Skill。

自动记忆的风险

如果系统自动把所有对话都提取成记忆，会把临时偏好当成永久偏好，把错误路径保存，把未完成任务当成事实。

对于技术工作，宁可少记一点，也不要把未验证信息长期保存。

团队场景里的 Memory

团队环境要考虑 Memory 是个人级还是团队级，谁可以写入，谁可以读取，是否包含敏感信息，是否需要审计。

不要把密钥、隐私信息或敏感业务数据随意写入长期记忆。

常见失败点

  • 记住了临时状态；
  • 记住了错误结论；
  • 记忆过多干扰上下文；
  • 用 Memory 替代当前检查。

完成标准

读完后，你应该能分清什么信息适合长期记忆，什么只属于当前任务，以及为什么错误记忆比没有记忆更危险。
