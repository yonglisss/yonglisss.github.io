如果你觉得很多 AI 产品还停留在“能聊，但不真正干活”，那今天这个 GitHub 热门项目值得看。Agent-Native 想解决的不是再做一个聊天壳，而是把 Agent、UI、动作能力和应用状态 放进同一套框架里：同一个动作既能被用户点击触发，也能被 Agent 调用，还能通过 HTTP、CLI、MCP 和 A2A 暴露出去。

📌 它适合谁

  • 想做 AI 原生应用，而不只是做一个聊天窗口的团队
  • 需要 Agent + 界面 + 数据状态 一起工作的开发者
  • 想把 Claude、Codex、Cursor 这类外部 Agent 接进自己产品的人

🔍 为什么值得关注

第一，它强调 “动作定义一次，多处复用”。官方 README 明确写到，一个 action 可以同时服务 UI、Agent、API、MCP、A2A 和 CLI，这比为每个入口分别接一层逻辑更像真正的产品工程。

第二，它不是只讲概念。官方文档已经给出多种落地形态：Headless agent、Rich chat、Embedded sidecar、Full application，还提供 Calendar、Content、Analytics、Slides 等模板，说明它在往“可直接做产品”走，而不是停在 demo。

第三，它对外部生态比较开放。官方文档写明，应用可通过远程 MCP 连接 Claude、ChatGPT、Codex、Cursor、GitHub Copilot 等宿主；这意味着你做的不是孤立 Agent，而是一个能接入现有 AI 工作流的应用层。

⚡ 怎么开始

最短路径不是先啃全套概念，而是直接看模板：

npx @agent-native/core@latest create my-platform
cd my-platform
pnpm install
pnpm dev

如果你只想先验证思路，也可以走 headless 模式，先定义一个 action，再让 Agent、CLI 或 MCP 去调用它。

💡 使用提醒

  • 这个项目更适合 有产品化目标 的团队；如果你只是给现有网站加一个简单问答框，它可能偏重。
  • 它把 Agent、状态、权限、协作、连接器揉在一起，能力很强，但也意味着学习曲线不会太低。
  • GitHub release 更新频率很高，适合持续关注，但正式用于生产前，最好先评估版本稳定性和数据库/部署约束。

结论

如果你关心的不是“再接一个模型”，而是 怎样把 Agent 做成真正可操作、可协作、可嵌入业务流程的应用，Agent-Native 很值得加入观察名单。它未必适合所有人，但对想做 AI 产品而不是 AI 玩具的团队，参考价值不低。
