Technical Guide

14. 实战一：用 DeerFlow 做一次深度研究

用一个研究任务串起模型、搜索、抓取、来源检查和报告生成，理解 DeerFlow 如何完成资料型任务。

实战目标

这一篇不追求复杂功能，只做一个典型任务：让 DeerFlow 完成一次深度研究，并产出结构化报告。

示例主题可以是：

请研究 DeerFlow 2.0 和普通聊天机器人的区别，要求基于官方资料，输出一份 Markdown 报告。

这个任务会用到模型、搜索、抓取、来源判断、报告写作和文件产物，适合检查 DeerFlow 的主链路。

前置条件

先确认：

  • DeerFlow 已能启动；
  • 至少有一个可用模型；
  • 搜索 provider 已配置；
  • 如果要保存文件，文件写入或 sandbox 能力可用；
  • make doctor 没有阻塞错误。

如果搜索还没配置，可以先把任务改成“基于我提供的资料”，不要让模型凭空研究。

任务提示词

可以从这个提示词开始：

请基于 DeerFlow 官方 README、配置文档和官网资料，研究 DeerFlow 2.0 和普通聊天机器人的区别。
要求：
1. 不要只基于搜索摘要写结论，先读取官方原文。
2. 报告结构包括：背景、核心差异、关键能力、适合场景、使用风险、下一步学习路线。
3. 每个关键判断后标注依据来自哪个官方资料。
4. 最终输出 Markdown。

这个提示词的重点是：要求读取原文、要求标注来源、要求结构化输出。

观察执行过程

运行时重点看几个点：

  • 是否真的调用了搜索或抓取工具；
  • 是否读取了官方 README 或文档；
  • 是否只停留在搜索摘要；
  • 是否出现来源不明的判断；
  • 是否生成了清晰报告结构。

如果 Agent 没有读取原文，就不要信最终报告。

检查输出质量

一份合格报告至少要满足：

  • 关键事实来自官方资料；
  • 不把营销口号当结论；
  • 能解释 DeerFlow 的工程价值；
  • 能区分模型、工具、sandbox、skills、sub-agents；
  • 对限制和风险有说明。

如果要保存为文件

可以在提示词里加：

请把最终报告保存为 deerflow-research-report.md。

然后验证文件是否真的存在、内容是否完整、路径是否符合预期、是否能在前端 artifacts 里看到。

不要只相信“已保存”的文字回复。

常见失败点

  • 搜索不可用：普通问答能跑，研究任务失败，通常是搜索 provider 没配好。
  • 来源质量差：优先要求官方 README、官网和配置文档。
  • 报告太空：提示词里要要求结构、来源和边界。
  • 文件没生成：检查文件工具、sandbox 目录、artifact 展示和实际路径。

复盘

这类研究任务的核心不是让模型写得漂亮，而是让它先拿到可靠材料，再基于材料整理。

DeerFlow 的价值在这里体现为：能把搜索、抓取、整理、写作、落盘放到同一条任务链路里。
