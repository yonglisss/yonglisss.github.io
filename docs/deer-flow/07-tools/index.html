Technical Guide

07. Tools 原理：Agent 为什么需要工具

从 DeerFlow 的工具目录和社区工具生态出发，理解 Tool、MCP、Skill 的边界，以及工具调用为什么必须配合 sandbox 和验证。

Tools 是 Agent 从“会说”到“会做”的第一步

模型本身只能生成文本。

如果一个 Agent 只能聊天，它最多给你建议；如果它能调用工具，它才可能完成任务：搜索资料、读取文件、执行命令、生成报告、调用外部系统。

所以从这一篇开始进入第一个能力模块：Tools。

这一篇先讲原理。后面可以继续接：

Tools 原理 → Tools 源码 → 自定义 Tool 实战

DeerFlow 里的工具大概分三类

基于官方当前仓库结构，工具相关代码主要分布在：

backend/packages/harness/deerflow/tools/
backend/packages/harness/deerflow/community/
backend/packages/harness/deerflow/mcp/

可以先这样理解：

目录更像什么解决什么问题
tools/内置工具系统工具定义、内置工具、工具同步、工具类型
community/第三方工具集合搜索、抓取、浏览器、图片搜索等 provider
mcp/MCP 工具接入把外部 MCP server 暴露的工具接进 Agent

这三个目录共同决定 DeerFlow 能对外部世界做什么。

Tool 解决的是动作问题

Tool 的核心职责是：把一个确定动作封装成模型可调用接口。

比如：

  • 搜索网页；
  • 抓取网页正文；
  • 查看图片；
  • 展示文件；
  • 管理 skill；
  • 调用子 Agent；
  • 通过 MCP 调外部服务；
  • 执行某个 sandbox 内动作。

Tool 不是“更长的 prompt”。它必须能执行一个明确动作，并返回可被 Agent 继续使用的结果。

搜索、抓取、浏览器是三种能力

研究类 Agent 最容易混淆三件事：

search：找候选来源
fetch / crawl：读取具体内容
browser：和网页进行更复杂交互

搜索结果通常只有标题、链接和摘要，不能直接当全文证据。严肃内容至少要进入 fetch / crawl 阶段，确认原文确实支持结论。

DeerFlow 的 community/ 目录里可以看到多种 provider：

brave
browserless
ddg_search
exa
fastcrw
firecrawl
groundroute
image_search
infoquest
jina_ai
searxng
serper
tavily

这说明 DeerFlow 的搜索和抓取能力不是硬编码到一个服务里，而是通过 provider 方式扩展。

内置工具承担基础动作

deerflow/tools/builtins/ 里有一些基础工具，例如：

clarification_tool.py
invoke_acp_agent_tool.py
present_file_tool.py
setup_agent_tool.py
task_tool.py
tool_search.py
update_agent_tool.py
view_image_tool.py

从名字就能看出，它们不是单纯“联网搜索”。它们覆盖的是 Agent 运行过程中的基础动作：

  • 请求澄清；
  • 查看图片；
  • 展示文件；
  • 创建或更新 Agent；
  • 调用任务工具；
  • 查找可用工具；
  • 调用 ACP Agent。

这类工具更接近 Agent runtime 的基础设施。

MCP 是外部工具协议，不是普通内置工具

MCP 的价值在于：把外部系统以标准协议暴露给 Agent。

DeerFlow 当前有独立的 deerflow/mcp/ 目录：

cache.py
client.py
oauth.py
session_pool.py
tools.py

这说明 MCP 不只是“多一个工具函数”。它还涉及：

  • MCP client；
  • session pool；
  • OAuth；
  • tool cache；
  • MCP tools 转换。

所以接 MCP 时，排错层级也更多：

MCP server 是否启动？
连接配置是否正确？
OAuth / token 是否有效？
工具 schema 是否能被加载？
Agent 是否看得到这些工具？
工具调用是否返回了可用结果？

后面 MCP 模块会单独讲。

Tool、Skill、MCP 不要混用

这三者很容易被混在一起。

可以用这张表判断：

需求更适合
执行一个确定动作Tool
复用一套工作方法Skill
接入外部系统或服务MCP
临时告诉模型这次怎么做Prompt

例子：

  • “读取 README 文件”是 Tool。
  • “写教程前必须核对官方 README，并跑 build 验证”是 Skill。
  • “调用公司内部知识库搜索”更适合 MCP。
  • “这次文章写短一点”是 Prompt。

工具越强，越需要边界

工具能力越强，风险越高。

比如：

  • 搜索工具可能带来低质量来源；
  • 抓取工具可能失败或拿不到正文；
  • 文件工具可能写错路径；
  • bash 工具可能删除文件、泄露密钥、卡住进程；
  • MCP 工具可能访问外部敏感系统。

所以 DeerFlow 的工具体系必须和这些机制一起看：

config
sandbox
guardrails
permissions
thread workspace
artifacts
tracing

只看 tool 定义，不看执行边界，是不完整的。

工具结果必须验证

工具调用成功，不等于任务成功。

常见失败场景：

  • 搜索返回了结果，但来源不可靠；
  • 抓取返回了正文，但正文和问题无关；
  • 文件写入成功，但路径错了；
  • bash 返回 0，但实际没有产生预期文件；
  • MCP 工具返回结构化数据，但 Agent 理解错了字段；
  • 子工具调用过多，导致上下文被噪音淹没。

所以开发者做二开时，要给关键工具设计“可验证输出”。

比如：

写文件后：读取文件确认内容
跑命令后：检查退出码和关键输出
搜索后：抓取原文而不是只看摘要
生成报告后：检查文件是否存在、格式是否正确

这也是 Agent 工程和普通 prompt 工程的区别。

什么时候需要自定义 Tool

不是所有需求都要写 Tool。

适合自定义 Tool 的情况：

  • 有一个动作会频繁执行；
  • 输入输出比较稳定；
  • 需要访问外部系统；
  • 需要封装权限和错误处理；
  • 用 prompt 描述太不稳定；
  • 用 bash 临时拼命令太危险。

不适合自定义 Tool 的情况：

  • 只是一次性任务；
  • 只是文字风格要求；
  • 流程还没稳定；
  • 输入输出没有边界；
  • 用 Skill 就能表达清楚。

这一篇的完成标准

读完这一篇，你应该能分清：

  • Tool 负责执行动作；
  • Skill 负责复用方法；
  • MCP 负责接外部系统；
  • Search / fetch / browser 是不同层次；
  • 工具调用必须配合 sandbox、权限和验证；
  • 自定义 Tool 应该封装稳定、高频、边界清楚的动作。
