Technical Guide

07. Tools：内置工具和自定义工具

理解 DeerFlow 的工具体系：搜索、抓取、文件、bash、图片、MCP，以及什么时候该做成 tool。

为什么先讲 Tools

Agent 真正开始“做事”，通常是从工具调用开始的。

没有 Tools，模型只能在聊天窗口里生成文字。有了 Tools，它才可以搜索资料、读取文件、写入产物、执行命令、调用外部服务。

所以理解 DeerFlow，不能只看模型和 prompt。你要看它开放了哪些工具、工具在哪里执行、权限怎么控制、失败后怎么排查。

Tool 解决什么问题

Tool 解决的是“模型不能直接接触外部世界”的问题。

模型本身不知道你的本地文件，也不能真的访问网页，更不能直接执行 bash。系统需要把这些能力封装成可调用接口，再让模型在合适的时候选择调用。

常见能力包括：

  • Web search：搜索网页；
  • Web fetch / crawl：抓取网页正文；
  • File operations：读取、写入、替换文件；
  • Bash execution：执行命令；
  • Image / upload tools：处理图片和上传文件；
  • MCP tools：通过 MCP server 接入外部系统。

一句话：Tool 是 Agent 的手。

搜索和抓取不是一回事

搜索结果通常只有标题、链接和摘要。做严肃内容时，不能只基于摘要写结论。你需要抓取原文，确认内容确实支持你的判断。

所以研究类任务最好分两步：先 search 找候选来源，再 fetch/crawl 读正文。只看摘要就写完整报告，是很多 Agent 研究任务的常见错误。

文件工具：从回答变成产物

文件工具让 DeerFlow 不只是在聊天框里输出答案。

它可以读取项目配置、写 Markdown 报告、修改源码、替换文本、保存中间结果。文件能力是 Agent 工程化的关键，因为真实交付物通常应该是可提交、可复用的文件。

但文件能力也最容易出问题：写错路径、覆盖旧文件、生成了文件但前端没展示、在错误工作目录执行。文件操作后，要检查文件存在、读取内容、运行构建或测试。

Bash 工具：强，但危险

bash 可以安装依赖、运行测试、执行脚本，也可能删除文件、泄露密钥、安装不可信依赖、长时间占用资源。

所以 bash 不应该无限开放。它需要配合 sandbox、权限控制和任务边界。入门时先跑低风险命令，明确工作目录，修改文件前看 diff，执行长命令前确认资源成本。

MCP 工具：把外部系统接进来

MCP 可以把数据库、GitHub、文档系统、内部 API、浏览器自动化等能力暴露给 Agent。

MCP 的价值是标准化工具接入。但它也带来新的排错层：server 是否启动、配置是否正确、token 是否有效、工具 schema 是否能被模型理解。

建议顺序是：先跑通内置工具，再接一个简单 MCP server，最后接复杂内部系统。

Tool、Skill、MCP 怎么选

需求更适合
按这个流程做事Skill
执行一个确定动作Tool
接入一个外部服务MCP
临时告诉模型这次怎么写Prompt

比如：“写技术教程前必须读官方 README”是 Skill；“读取 README 文件”是 Tool；“调用公司内部知识库搜索”是 MCP。

常见失败点

  • 工具没有启用：模型说不能搜索、不能写文件，优先查配置和权限。
  • 工具返回空：搜索为空、抓取失败、文件不存在，模型不能凭空补。
  • 工具调用太多：一直搜索、一直读文件，需要限制步骤并阶段性总结。
  • 工具结果没被正确使用：工具返回正确，但模型总结错了，需要更好的上下文组织或 skill。

完成标准

读完后，你应该能判断一个能力应该做成 Tool、Skill 还是 MCP，也知道搜索、文件、bash 为什么必须验证。
