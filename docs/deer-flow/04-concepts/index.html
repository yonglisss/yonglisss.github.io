Technical Guide

04. 整体架构：DeerFlow 到底在调度什么

从官方后端架构图出发，理解 DeerFlow 的核心组件：Gateway、Lead Agent、middleware、tools、skills、sub-agents、memory、sandbox 和 artifacts。

这一篇先画地图

从这一篇开始，教程进入“架构地图”。

不要急着读每个源码文件。先把 DeerFlow 的主链路看清楚：请求从哪里进来，谁负责调度，工具怎么接入，sandbox 和 memory 在哪里发挥作用，最后结果怎么返回。

官方后端 README 给出的架构可以简化成这样：

用户 / 前端 / IM 入口
  ↓
Nginx
  ↓
Gateway API（FastAPI + agent runtime）
  ↓
Lead Agent
  ↓
Middleware / Tools / Sub-agents / Memory / Sandbox
  ↓
Artifacts / Streaming / Final Response

这张图比单独背概念更重要。后面读源码时，每个模块都要放回这条链路里看。

DeerFlow 不是一个单点 Agent

官方 README 把 DeerFlow 2.0 定义为 super agent harness。这里的重点是 harness：它不是只提供一个 Agent 函数，而是把 Agent 运行需要的工程能力组装起来。

官方后端 README 也明确说，DeerFlow backend 是一个基于 LangGraph 的 AI super agent，带有 sandbox execution、persistent memory 和 extensible tool integration。

所以 DeerFlow 的重点不是“有几个 Agent”，而是这些能力如何协同：

  • 请求入口和路由；
  • 线程、运行、上传、产物管理；
  • 模型选择；
  • middleware 链；
  • tools / MCP / community tools；
  • skills 注入；
  • sub-agent delegation；
  • sandbox 执行；
  • memory 上下文；
  • streaming 返回。

这也是为什么它更像一个 Agent 运行底座，而不是一个聊天页面。

第一层：入口和 Gateway

官方架构里，Nginx 位于最外层，负责统一反向代理。

常见请求会被分到几类：

/api/langgraph/*  → Gateway 的 LangGraph-compatible API
/api/*            → Gateway API 的其他接口
/                 → 前端页面

这说明 DeerFlow 对外不是只有网页。它同时要处理：

  • 前端发起的对话和任务；
  • LangGraph 兼容接口；
  • models、MCP、skills、memory、artifacts、uploads 等管理接口；
  • 后续可能接入的 IM channels。

所以看 DeerFlow 架构时，第一层不是 Agent，而是入口层：请求先进入 Gateway，再进入 Agent runtime。

第二层：Lead Agent 是运行入口

官方后端 README 里提到，Lead Agent 是 runtime entry point，由 make_lead_agent(config) 创建。

可以先把 Lead Agent 理解成“主控”。它负责把用户任务变成一次可执行的 run，并在过程中调用不同能力。

它不是单独工作，而是组合了几类东西：

  • 动态模型选择；
  • middleware chain；
  • tools；
  • sub-agent delegation；
  • system prompt；
  • skills 注入；
  • memory context；
  • working directory guidance。

所以后面读源码时，不要只盯着“模型调用”本身。真正重要的是 Lead Agent 周围这一圈运行机制。

第三层：Middleware 处理横切逻辑

DeerFlow 后端 README 提到 middleware chain，而且是按严格顺序执行。

这类设计很关键。因为 Agent 任务不是一次普通函数调用，它需要在运行前后处理很多横切逻辑：

  • 为每个 thread 创建隔离目录；
  • 处理上传文件；
  • 准备工作目录；
  • 注入 memory；
  • 控制 token budget；
  • 处理 artifacts；
  • 清理运行状态；
  • 影响工具和模型上下文。

如果没有 middleware，所有逻辑都会堆到 Agent 主流程里，很快失控。

你可以把 middleware 理解成：在 Agent 真正推理前后，对运行环境和上下文做整理的人。

第四层：Tools / Skills / Sub-agents

这三类能力经常被混在一起，但它们解决的问题不同。

能力解决什么问题更像什么
Tools让 Agent 执行具体动作手和工具箱
Skills让 Agent 复用稳定方法SOP / 工作流经验
Sub-agents把复杂任务拆出去临时协作同事

举个例子：

  • 搜索网页、读文件、执行命令，是 tool。
  • “写技术教程时先查官方文档、再验证构建”，是 skill。
  • “让一个子 Agent 查源码，一个子 Agent 查配置文档”，是 sub-agent delegation。

后面教程会按这个顺序拆：每个能力先讲原理，再看源码，再做一个小实战。

第五层：Sandbox 和文件产物

DeerFlow 不是只输出文字。它要执行代码、读写文件、生成报告、处理上传内容。

只要 Agent 能执行，就必须有边界。Sandbox 的作用就是把执行环境隔离出来，降低误改宿主机、依赖污染和多任务互相影响的风险。

官方后端 README 也强调 per-thread environments。也就是说，每个 thread / run 不是随便混在一起跑，而是要有自己的工作目录、上传、输出和产物。

从使用者角度看，sandbox 和 artifacts 最容易影响这些体验：

  • 文件到底写到哪里；
  • 上传文件能否被 Agent 看到；
  • 生成报告能否在前端下载；
  • bash / 文件工具是否有权限；
  • Docker sandbox 第一次启动为什么慢。

这部分后面会单独拆。

第六层：Memory 和上下文

Memory 不是聊天记录仓库。它解决的是长期上下文问题。

在 DeerFlow 这类系统里，memory 通常会和 system prompt、skills、thread data、uploads、tool results 一起影响 Agent 当前看到的信息。

所以你可以把一次任务的上下文来源看成：

用户输入
+ 当前 thread 状态
+ 上传文件
+ skill 指令
+ memory context
+ tool results
+ working directory 信息

Agent 效果不稳定，很多时候不是“模型突然变笨”，而是这些上下文没有组织好。

一个简化执行链路

先记住这个版本：

用户提交任务
  ↓
Gateway 创建 thread / run
  ↓
Middleware 准备目录、上传、memory、上下文
  ↓
Lead Agent 选择模型并开始推理
  ↓
按需调用 tools / MCP / sub-agents
  ↓
需要执行时进入 sandbox / 文件系统
  ↓
产物进入 artifacts，结果通过 streaming 返回

这不是完整源码级链路，但足够作为阅读地图。

这一篇的完成标准

读完这一篇，你不需要记住所有模块名。

只要能回答这几个问题就够：

  • 请求不是直接进模型，而是先经过 Gateway 和运行管理。
  • Lead Agent 是主控，但它依赖 middleware、tools、skills、memory 和 sandbox。
  • Tools、Skills、Sub-agents 不是一回事。
  • Sandbox 和 artifacts 决定了执行边界和任务产物。
  • 后面读源码时，要把每个文件放回“入口 → runtime → Agent → tool/sandbox/artifact”这条链路里。
