Technical Guide

12. MCP：接入外部工具和服务

理解 MCP 的作用：什么时候用 MCP、如何配置、如何验证工具可见，以及常见连接和权限问题。

MCP 解决的是工具接入问题

Agent 要真正有用，迟早要接外部系统：GitHub、数据库、文档系统、浏览器、内部 API、云服务、企业知识库。

如果每接一个系统都写一套专用适配，成本会很高。MCP 的价值，是用相对统一的协议把外部能力暴露给 Agent。

MCP Server 是什么

可以先把 MCP Server 理解成“工具服务”。

它对外提供一组工具，Agent 通过 MCP Client 发现并调用这些工具。

例如 GitHub MCP Server 可能提供搜索 issue、读取 PR、创建 comment、查询 workflow、获取文件内容等能力。

MCP 和内置 Tool 的区别

类型适合场景
内置 Tool通用能力，如文件、bash、搜索、图片
Community Tool常见第三方能力，如 Tavily、Firecrawl
MCP Tool外部服务、企业系统、可独立运行的工具服务

如果某个能力只在 DeerFlow 内使用，写内置 tool 可能更直接。如果希望多个 Agent 系统都能调用，做成 MCP server 更有复用价值。

什么时候应该用 MCP

适合用 MCP 的场景：已有现成 MCP server、工具需要独立部署、服务需要自己的认证和权限、未来可能被多个 Agent 使用、工具输入输出结构化。

不适合的场景：一次性脚本、功能很简单、schema 不稳定、服务无法长期运行、权限模型没设计好。

配置 MCP 的基本思路

具体字段以 DeerFlow 当前官方配置为准。一般会涉及 server 名称、启动方式或 URL、传输方式、token / OAuth、是否启用、工具发现和缓存。

本地 stdio server 重点看启动命令。HTTP / SSE server 重点看 URL、认证和网络可达性。

验证 MCP 是否接入成功

不要一接完就跑复杂任务。

分层验证：server 能否单独启动、DeerFlow 能否连接、工具列表是否可见、schema 是否正常、用简单参数调用只读工具、再让 Agent 在真实任务中使用。

如果工具列表不可见，问题不在模型，而在 MCP 配置或连接。

工具命名很重要

模型选择工具时，会看工具名和描述。

不好的命名：do_thing、query、run。

更好的命名：search_github_issues、read_notion_page、query_customer_orders。

工具描述要说明做什么、输入字段、返回什么、限制是什么、是否有副作用。

权限和副作用

MCP 工具可能创建 issue、更新数据库、发送消息、部署服务。

只读工具可以较宽松；写入工具需要确认或权限；删除、发布、部署工具必须有强约束和审计。

MCP 和 Skills 配合

MCP 提供能力，Skill 提供用法。

接了 GitHub MCP，最好再有 GitHub 工作流 skill，告诉 Agent 查 issue 前先确认 repo、push 后检查 CI、不要把 PR 创建当成完成交付。

完成标准

读完后，你应该知道 MCP 是外部工具接入层，server 要先独立验证，工具 schema 和权限会直接影响 Agent 可用性。
