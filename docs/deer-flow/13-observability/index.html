Technical Guide

13. 可观测性：LangSmith / Langfuse 追踪

讲清 Agent 为什么需要 tracing，如何观察模型调用、工具链路、成本、延迟和失败原因。

Agent 不可观测，就很难排错

普通接口失败，通常看日志、状态码、异常栈就能定位。

Agent 任务失败更复杂。它可能失败在模型理解、工具选择、搜索质量、sandbox 执行、子 Agent 汇总、上下文长度、token 成本等很多层。

所以 Agent 系统必须有可观测性。否则你只能看到“结果不对”，却不知道是哪一步不对。

Tracing 解决什么问题

Tracing 可以把一次任务拆开看：每次模型调用输入了什么，输出了什么，调用了哪些工具，工具参数和返回是什么，哪一步最慢，哪一步 token 最多，失败发生在哪一层。

这对调 Agent 很关键。

DeerFlow 支持哪些 tracing

官方 README 提到 DeerFlow 支持 LangSmith 和 Langfuse 这类 tracing provider。

具体配置以 config.example.yaml 和官方文档为准。通常会涉及是否启用 tracing、provider、API key、project / host、环境变量。

入门阶段不一定马上配置 tracing，但一旦开始做复杂任务、团队服务或二开，它会很有用。

先看哪些指标

模型调用

看模型实际收到了什么。很多问题不是模型不会，而是上下文里没有关键信息，或塞了太多无关内容。

工具调用

看 Agent 调用了哪些工具，参数是否正确。它应该读官方 README，却只看了搜索摘要，这种问题 tracing 能看到。

工具返回

工具返回空、超时、被截断，都会影响结果。不要只看模型输出，要看工具是否提供了有效输入。

Token 和成本

长任务很容易成本失控。如果一次任务反复调用大模型、抓取长网页，就需要优化流程或限制 token budget。

延迟

看慢在哪里：模型慢、搜索慢、sandbox 慢，还是前端流式返回慢。

如何排查一次失败

按顺序看：用户原始输入是否清楚，Lead Agent 第一次判断是否正确，是否选对工具，工具参数是否正确，工具返回是否有效，后续模型是否正确使用工具结果，子 Agent 是否跑偏，最终汇总是否丢约束。

不要只看最后一条输出。Agent 的错误往往发生在中间。

日志和 tracing 的区别

日志偏系统层：服务是否启动、请求是否失败、异常是什么。

Tracing 偏任务链路：模型和工具之间发生了什么。

排查服务启动失败，看日志；排查 Agent 为什么答错，看 tracing。

隐私和安全

Tracing 会记录大量输入输出，可能包含 API key、用户隐私、公司内部文档。

不要为了调试，把敏感数据无脑送到外部 tracing 平台。团队环境要考虑脱敏、权限和数据保留周期。

Token Budget

DeerFlow 配置里有 token budget 相关能力，可以设定单次 run 最大 token，到达阈值时提醒，超过硬限制时强制收束。

这对长期任务很重要。没有预算控制，Agent 可能在搜索、总结、重试中消耗大量 token。

完成标准

读完后，你应该知道 Agent 排错不能只看最终答案，tracing 能看到模型调用、工具调用和中间结果，也知道 tracing 数据可能包含敏感信息。
