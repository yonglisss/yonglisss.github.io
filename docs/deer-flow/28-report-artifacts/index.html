Technical Guide

28. 生成报告和文件产物

让研究型 Agent 不只返回一段话，而是生成可保存、可检查、可下载的 Markdown 报告文件。

最终输出不要只是一段回复

研究型 Agent 的结果应该是产物。

最简单的产物就是 Markdown 报告。

它应该能被保存、读取、下载或继续编辑。

报告结构

建议第一版固定结构：

标题
研究问题
结论摘要
证据和来源
详细分析
不确定性
下一步建议

固定结构方便验收，也方便后续自动化处理。

文件路径

文件保存要有明确路径。

不要让 Agent 随便起一个临时位置。

建议路径包含：

日期
任务主题
报告类型

这样后面容易检索和归档。

保存后要验证

生成文件后，必须确认：

文件存在
内容非空
Markdown 可读
路径返回给用户

如果有下载或展示工具，也要确认前端能看到。

这一篇你要记住

报告不是最后一段回答。

报告应该是可检查的文件产物。否则“研究型 Agent”很难进入真实工作流。

Markdown 为什么适合作为第一版产物

Markdown 足够简单，也足够通用。

它可以被网页展示，可以进知识库，也可以后续转 PDF 或同步到 Git。

第一版不要直接追求复杂格式。

文件命名建议

可以采用：

YYYY-MM-DD-topic-research.md

如果主题很长，先做 slug 化。

不要让 Agent 生成随机文件名。

报告里的来源格式

建议统一：

- 标题：...
  URL：...
  用途：支撑哪个判断

来源不是装饰，它要能反查。

验收时打开文件

文件写完后，最好读取前几行确认内容。

只检查“路径存在”还不够。
