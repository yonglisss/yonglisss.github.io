Technical Guide

08. Skills 原理：把经验沉淀成可复用能力

理解 DeerFlow Skill 的定位：它不是更长的 prompt，也不是 tool，而是可安装、可解析、可校验、会影响 Agent 行为的工作方法。

Skill 不是更长的 Prompt

Prompt 更像临时指令。Skill 更像可复用的工作方法。

如果你经常让 Agent 做同一类任务，比如：

  • 写开发者教程；
  • 做深度研究；
  • 生成 newsletter；
  • 设计前端页面；
  • 做代码文档；
  • 生成 PPT；
  • 评审论文；
  • 部署到 Vercel；
  • 把 Claude 风格 skill 转成 DeerFlow skill；

这些都不应该每次靠 prompt 从零描述。更好的方式是沉淀成 Skill。

官方仓库当前 skills/public/ 下可以看到很多公开 skill，例如：

这说明 Skill 在 DeerFlow 里不是装饰品，而是重要能力层。

它让 Agent 不只是“会调用工具”，还知道某类任务应该怎么做。

Skill 解决什么问题

Agent 失败有时不是模型不够强，而是每次都从零开始。

比如写一篇开发者教程，需要一套稳定流程：

先查官方 README
再看配置示例
确认命令是否真实存在
按可执行路径组织文章
写完跑 build
只提交目标文件
不要脑补没查到的信息

如果每次都写在 prompt 里，很容易漏。

Skill 的作用就是把这类稳定流程固化下来，让 Agent 在相关任务中自动带上这些约束。

Skill、Tool、Memory 的边界

这几个概念要分清：

类型解决的问题示例
Prompt这一次怎么做这篇写短一点
Skill同类任务以后怎么做写教程先查官方资料，写完跑 build
Tool执行一个动作读取文件、搜索网页、执行命令
MCP接入外部系统GitHub、数据库、文档系统
Memory保存长期事实用户偏好、项目固定路径

Skill 通常不直接执行动作。它会影响 Agent 如何选择步骤、调用工具、判断完成标准。

一句话：

Tool 是手，Skill 是做事方法，Memory 是长期事实。

Skill 在源码里不是一份普通 Markdown

官方仓库里，Skill 相关实现主要在：

backend/packages/harness/deerflow/skills/
├── installer.py
├── parser.py
├── permissions.py
├── security_scanner.py
├── slash.py
├── storage/
├── tool_policy.py
├── types.py
└── validation.py

从文件名就能看出，Skill 不只是“读一个 Markdown”：

  • parser.py：解析 Skill 内容；
  • installer.py：安装或引入 Skill；
  • validation.py：校验 Skill 是否合规；
  • permissions.py：处理权限；
  • security_scanner.py：扫描潜在风险；
  • tool_policy.py：约束 Skill 能用哪些工具；
  • storage/：Skill 的存储抽象；
  • slash.py：可能和 slash command 形式的调用有关。

这就是为什么 Skill 是一个工程能力，而不是简单 prompt 模板。

Skill 会影响 Agent 的上下文和工具边界

一个好 Skill 至少会影响三件事：

  1. 任务流程
    让 Agent 知道这类任务应该按什么步骤做。

  2. 完成标准
    让 Agent 知道什么时候可以停止，什么时候还要验证。

  3. 工具策略
    某些 Skill 可能会限制或引导 Agent 使用哪些工具。

例如一个“深度研究” Skill，可能要求：

  • 先搜索多个来源；
  • 再抓取原文；
  • 区分事实和推断；
  • 标注来源；
  • 不基于摘要写结论。

这比“请认真研究”有用得多。

什么适合做成 Skill

适合沉淀成 Skill 的任务通常有这些特征：

  • 重复出现；
  • 步骤相对稳定；
  • 容易踩坑；
  • 需要验证；
  • 能明显减少沟通；
  • 不是一次性信息；
  • 不是很快过期的数据。

比如：

  • 开发者教程写作；
  • 深度研究报告；
  • 代码文档生成；
  • 数据分析报告；
  • 前端设计规范；
  • Vercel 部署流程；
  • GitHub 项目调研；
  • 学术论文评审。

这些都很适合 Skill。

什么不适合做成 Skill

不适合做成 Skill 的东西：

  • 某次任务进度；
  • 某个临时链接；
  • 某个 commit hash；
  • 某个 PR 编号；
  • 一次性数据；
  • 没验证过的猜测；
  • 单纯的写作口号。

比如“文章要有深度、逻辑清楚”不是好 Skill。它太抽象，不能指导 Agent 行动。

更好的写法是：

写开发者教程时：
1. 先读取官方 README、安装文档和配置示例；
2. 所有命令必须来自官方文档或当前仓库；
3. 正文按“问题 → 步骤 → 验证 → 常见坑”组织；
4. 写完运行真实 build；
5. 官方资料拿不到时停止，不要脑补。

这才是能复用的流程。

Skill 也需要权限和安全边界

Skill 可以强烈影响 Agent 行为，所以必须有边界。

如果一个 Skill 里写了“执行这个脚本”“使用这个外部服务”“读取这个目录”，系统就需要考虑：

  • 这个 Skill 是否可信；
  • 是否允许它使用相关工具；
  • 是否包含危险命令；
  • 是否会访问敏感文件；
  • 是否会泄露密钥；
  • 是否应该限制它的 tool policy。

这也是官方代码里会有 permissions.py、security_scanner.py、tool_policy.py 的原因。

Skill 越强，越不能只当文档看。

Skill 和最终实战的关系

后面终局实战是“自己写一个研究型 Agent”。

这里 Skill 会承担一个关键角色：约束 Agent 的工作方法。

比如这个研究型 Agent 可以有一个 Skill：

research-report-skill

它规定：

  • 先明确研究问题；
  • 至少查多个来源；
  • 搜索后必须抓取正文；
  • 区分事实、观点、推断；
  • 输出 Markdown 报告；
  • 报告包含来源列表；
  • 生成后检查文件是否存在。

这样终局实战就不只是“调用几个工具”，而是有稳定工作流的 Agent。
