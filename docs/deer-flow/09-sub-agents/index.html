09. Sub-agents：多智能体是怎么分工的

理解子 Agent 的作用、适用场景、上下文隔离、结果汇总和常见失控点。

多 Agent 不是越多越好

多 Agent 更像把复杂任务拆给多个临时同事。拆得好，会加速；拆得差，会制造更多沟通成本。

学习 DeerFlow 的 Sub-agents，重点不是看能创建几个 Agent，而是看什么时候值得拆、怎么拆、怎么汇总。

Sub-agent 解决什么问题

主要解决三个问题：上下文隔离、专业分工、并行处理。

例如一次技术调研可以拆成：一个子 Agent 读官方 README，一个看项目结构，一个找部署文档，主 Agent 合并结论并写教程。

什么任务适合拆

适合拆的任务通常满足：信息源独立、子任务边界清楚、每个子任务有明确产出、主任务需要综合多个视角、单一上下文会太长。

例如大型代码库阅读、多方案技术选型、长报告资料收集、多模块排错、PR 代码审查。

什么任务不适合拆

改一行配置、写一个短说明、查一个确定命令、需求还不清楚、子任务之间强依赖，都不适合硬拆。

这类任务直接做更稳定。

子任务要怎么写

给子 Agent 的任务要具体。

不好的写法：

你去研究一下 DeerFlow。

更好的写法：

请只阅读 DeerFlow 官方 README 和 config.example.yaml，总结模型 provider、搜索、sandbox 三类配置的关键字段。不要写教程，只返回要点和来源文件。

好的子任务应该包括目标、范围、输入材料、输出格式、不要做什么、验证要求。

主 Agent 负责汇总

子 Agent 的输出不是最终答案。

主 Agent 必须去重、判断冲突、过滤无关信息、补齐遗漏、统一风格、给出最终结论。

如果只是把多个结果拼在一起，多 Agent 反而会降低质量。

常见失控点

  • 子任务太大：子 Agent 也会发散。
  • 子任务太模糊：输出无法直接使用。
  • 结果无法验证：没有来源、路径、命令输出。
  • 汇总阶段丢约束：最终结果不符合用户要求。

Sub-agent 和 Tool 的区别

Tool 是具体动作。Sub-agent 是带上下文的任务执行者。

读取文件是 Tool；让一个子 Agent 阅读三个文件并总结架构，是 Sub-agent。

Sub-agent 和 Skill 的关系

Skill 可以规定什么时候拆子 Agent、怎么拆、怎么汇总。Sub-agent 是流程中的执行角色。

完成标准

读完后，你应该知道多 Agent 不是默认更好，只有任务边界清楚、产出可验证时才值得拆。
