18. 部署与排错：从本地可用到长期运行

整理 DeerFlow 研究型 Agent 的部署和排错路径：配置、Docker、sandbox、MCP、tracing、artifacts、日志和回滚。

本地跑通不等于长期可用

make dev 或本地页面能跑起来，只说明开发环境可用。

如果要让研究型 Agent 长期服务自己或团队，还要考虑：

  • 配置管理；
  • 密钥管理；
  • Docker / 本地模式选择；
  • sandbox 边界；
  • MCP 稳定性；
  • tracing；
  • artifacts 存储；
  • 日志和排错；
  • 升级和回滚。

这一篇不是替代官方部署文档，而是给你一张长期运行检查表。

先确认部署模式

DeerFlow 常见运行方式可以分成：

本地开发模式
Docker / compose 模式
生产部署模式

选择时先问：

  • 只是自己本地试用，还是团队使用？
  • 是否需要执行 bash？
  • 是否处理不可信文件？
  • 是否需要 IM channels？
  • 是否需要公网访问？
  • 是否接 MCP 外部系统？
  • 是否需要 tracing 和持久化？

如果只是学习，本地模式足够。只要进入长期运行或团队使用，就应该优先考虑 Docker / 更明确的隔离边界。

配置不要散落

关键配置来自：

config.yaml
config.example.yaml
extensions_config.json
.env
DEER_FLOW_CONFIG_PATH
DEER_FLOW_EXTENSIONS_CONFIG_PATH

建议：

  • config.example.yaml 只当模板；
  • 本地实际配置放 config.yaml；
  • 密钥放 .env 或安全的 secret 管理；
  • 不要把真实 key 写进 Markdown、Skill、Git；
  • 修改配置后记录原因；
  • 部署前用 make doctor 或对应检查命令验证。

Sandbox 是部署边界核心

长期运行时，sandbox 是重点。

至少确认：

当前使用哪个 sandbox provider？
是否允许 host bash？
工作目录在哪里？
上传文件在哪里？
输出文件在哪里？
哪些目录被挂载？
是否有清理策略？

如果配置里使用 LocalSandboxProvider，要记住：它不是强隔离边界。官方配置模板也默认关闭 host bash。

团队环境里不要随便开启宿主机 bash。

MCP 要做健康检查

MCP 接入长期运行时，容易出现：

  • server 掉线；
  • token 过期；
  • 工具 schema 变化；
  • cache 未刷新；
  • 权限被收回；
  • 返回值变大；
  • 网络超时。

建议为每个 MCP 准备一个最小健康检查任务：

列出工具
执行一个只读查询
确认返回字段
确认 cache 状态

复杂研究任务失败时，先跑最小 MCP 调用，别直接排查整条链路。

Tracing 尽早打开

如果要长期使用，建议尽早配置 LangSmith 或 Langfuse。

官方后端 README 明确支持两者，并且可以同时启用。

Tracing 能帮助你看清：

  • 哪个模型被调用；
  • system prompt 里有什么；
  • skill 是否激活；
  • 工具调用参数；
  • 工具返回内容；
  • middleware 是否执行；
  • token 消耗；
  • 失败发生在哪一步。

没有 tracing，长任务排错会非常痛苦。

Artifacts 要能找回

研究型 Agent 的最终产物是报告文件。

部署时要确认：

  • 文件写入路径稳定；
  • artifacts 能被 Gateway 找到；
  • 前端能展示或下载；
  • 清理策略不会误删近期报告；
  • 文件名不会冲突；
  • 报告里没有泄露密钥；
  • 大文件不会塞爆上下文。

如果 artifacts 做不好，Agent 就会退化成聊天窗口输出。

排错顺序

遇到问题时，按这条线查：

1. 服务是否启动
2. 前端是否能访问
3. Gateway 是否可用
4. 配置是否加载正确
5. 模型 provider 是否可用
6. thread / run 是否创建
7. skill 是否激活
8. tool / MCP 是否可见
9. sandbox 是否能执行
10. artifacts 是否生成
11. streaming 是否返回
12. 前端是否展示

不要一上来就怀疑模型。

常见故障

现象优先检查
页面打不开前端、Nginx、端口、Docker compose
发送没反应Gateway、runs、streaming
模型不回复API key、provider、model name、网络
不调用工具tool 注册、tool_search、模型 tool call 支持
文件找不到sandbox 路径、artifacts、工作目录
MCP 不可用server、config、OAuth、cache
报告质量差Skill、来源抓取、证据表
任务卡住bash、MCP 超时、sub-agent、token budget

升级和回滚

Agent 系统升级要谨慎。

升级前：

  • 备份配置；
  • 记录当前 commit；
  • 导出或备份重要数据；
  • 确认官方 README 变更；
  • 先在测试环境启动；
  • 跑一条最小研究任务。

回滚时：

  • 回滚代码；
  • 回滚配置；
  • 清理不兼容缓存；
  • 检查数据库迁移；
  • 重新跑健康检查。

这一篇的完成标准

读完这一篇，你应该知道：

  • 本地跑通和长期运行的差别；
  • 配置、密钥、sandbox、MCP、tracing、artifacts 各自要检查什么；
  • 长任务失败时该按哪条链路排查；
  • 升级前后应该如何验证。
