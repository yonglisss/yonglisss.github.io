Technical Guide

04. 整体架构：DeerFlow 到底在调度什么

理解 Gateway、Lead Agent、runtime、middleware、tools、sandbox、MCP 和 streaming。

从官方架构看主线

官方后端 README 的架构图里，Nginx 在前面，Gateway API 位于核心位置。Gateway 提供 REST + agent runtime，承载 Models、MCP、Skills、Memory、Uploads、Artifacts、Threads、Runs、Streaming。

简化链路：

Frontend / API
  ↓
Nginx / Gateway
  ↓
Thread / Run
  ↓
Runtime
  ↓
Middleware Chain
  ↓
Lead Agent
  ↓
Tools / Sub-agents / Sandbox / MCP / Memory
  ↓
Artifacts / Streaming

Gateway 是入口

当前 Gateway routers 包括 models、mcp、skills、memory、uploads、artifacts、threads、runs 等。它不是普通转发层，而是统一后端入口。

Lead Agent 是决策中心

Lead Agent 在 backend/packages/harness/deerflow/agents/lead_agent/。它依赖 middleware 准备上下文、过滤工具、处理 memory、控制 token 和安全边界。

Runtime 管理运行

运行相关目录在 backend/packages/harness/deerflow/runtime/，包括 runs、events、store、stream_bridge、checkpointer。

能力模块

tools/       工具系统
skills/      可复用工作方法
subagents/   子 Agent 执行
sandbox/     执行和文件边界
mcp/         外部工具协议接入
memory/      长期上下文
tracing/     可观测性

完成标准

你能用一句话解释：DeerFlow 是以 Gateway 和 Lead Agent 为核心，把工具、记忆、沙盒、MCP、子 Agent 和文件产物组织成可运行任务流的 Agent harness。
