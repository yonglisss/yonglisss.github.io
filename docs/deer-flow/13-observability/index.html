Technical Guide

13. 可观测性：看清一次 Agent 为什么失败

理解 DeerFlow 的 tracing 与运行观测：LangSmith、Langfuse、metadata、run events、tool output budget 如何帮助定位 Agent 失败原因。

Agent 不可观测，就很难排错

普通接口失败，通常看状态码、日志、异常栈就能定位。

Agent 任务失败更复杂。它可能失败在：

  • 模型选择；
  • prompt 上下文；
  • memory 注入；
  • skill 激活；
  • tool schema；
  • MCP 连接；
  • sandbox 执行；
  • 文件路径；
  • artifacts 展示；
  • streaming 转发；
  • token budget；
  • 子 Agent 汇总。

如果没有观测能力，只看最终回答，很难知道真正原因。

DeerFlow 的 tracing 相关目录

官方当前仓库里 tracing 相关代码主要在：

backend/packages/harness/deerflow/tracing/
├── factory.py
└── metadata.py

后端 README 明确写到 DeerFlow 支持：

  • LangSmith Tracing；
  • Langfuse Tracing；
  • 两个 provider 同时启用。

当 tracing 开启时，可以观测 LLM calls、agent runs、tool executions、middleware processing 等过程。

LangSmith 配置

官方后端 README 给出的 LangSmith 配置大致是：

LANGSMITH_TRACING=true
LANGSMITH_ENDPOINT=https://api.smith.langchain.com
LANGSMITH_API_KEY=...
LANGSMITH_PROJECT=...

同时也兼容旧的 LangChain 变量，例如：

LANGCHAIN_TRACING_V2
LANGCHAIN_API_KEY
LANGCHAIN_PROJECT
LANGCHAIN_ENDPOINT

但如果同时存在，LANGSMITH_* 优先。

Langfuse 配置

官方后端 README 给出的 Langfuse 配置大致是：

LANGFUSE_TRACING=true
LANGFUSE_PUBLIC_KEY=...
LANGFUSE_SECRET_KEY=...
LANGFUSE_BASE_URL=https://cloud.langfuse.com

如果是自托管 Langfuse，就把 LANGFUSE_BASE_URL 改成自己的地址。

注意：不要把真实 key 写进教程或提交到仓库。文档中只能写占位符。

可以同时启用两个 provider

官方说明提到，如果 LangSmith 和 Langfuse 都启用，DeerFlow 会初始化并附加两个 callbacks，让同一份 run data 上报到两个系统。

这很适合迁移或对比阶段。

但也意味着：

  • 两边凭证都要正确；
  • 任一 provider 配错都可能影响初始化；
  • 观测数据会同时发送到两个平台；
  • 需要注意组织内部的数据合规要求。

可观测性要看什么

不要只看最终 answer。

一次 Agent run 至少要看这些层：

1. 用户输入和 thread / run id
2. system prompt 是否包含正确 skill / memory
3. 选用了哪个模型
4. 模型为什么选择某个 tool
5. tool 入参是否正确
6. tool 返回结果是否可用
7. sandbox 是否执行成功
8. 子 Agent 是否返回了结构化结果
9. artifacts 是否生成
10. 最终回答是否引用了真实结果

这能帮助你判断失败是“模型推理问题”，还是“工具/配置/上下文问题”。

Tool Output Budget 也属于可观测性

config.example.yaml 里有 Tool Output Budget Protection。

它解决的是：工具输出过大，撑爆模型上下文。

配置说明里提到，当工具结果超过阈值时，可以把完整输出持久化到磁盘，用一个紧凑 preview + 文件引用替代；模型后续可以再用 read_file 读取完整内容。

这对长网页、日志、大文件尤其重要。

否则一个工具结果就可能把上下文挤满，让模型忘掉用户目标。

运行事件和 persistence

DeerFlow 还有 persistence 目录，里面包含 run、run_event、thread_meta、feedback 等模块。

这意味着系统不仅关心最终答案，也关心运行过程数据。

典型价值：

  • 查询某次 run 的状态；
  • 追踪中间事件；
  • 收集反馈；
  • 记录 thread meta；
  • 排查某次执行为什么失败。

长期运行的 Agent 系统必须保留这类运行信息，否则很难运营和改进。

常见失败如何定位

1. Agent 答非所问

优先看：

  • system prompt；
  • skill 是否激活；
  • memory 是否污染；
  • 用户输入是否被正确传入；
  • 是否过早总结。

2. Agent 没调用工具

优先看：

  • 工具是否注册；
  • 模型是否支持 tool call；
  • tool_search 是否开启；
  • 工具 schema 是否太模糊；
  • prompt 是否给了明确动作目标。

3. 工具调用失败

优先看：

  • tool 入参；
  • API key / token；
  • MCP server 状态；
  • sandbox 权限；
  • 网络错误；
  • 返回值格式。

4. 文件没生成

优先看：

  • 文件工具是否执行；
  • sandbox 当前目录；
  • 写入路径；
  • artifacts 注册；
  • 前端展示。

5. 长任务卡住

优先看：

  • streaming 是否还在；
  • 子 Agent 是否未返回；
  • bash 是否卡住；
  • token budget 是否触发；
  • MCP 工具是否超时。

终局实战里怎么用可观测性

研究型 Agent 最终实战里，至少要观测这些点：

研究主题是否被正确解析
搜索工具返回了哪些来源
是否抓取了原文
是否调用了 MCP
是否使用了 Skill 约束输出
是否生成了 Markdown 文件
报告文件是否可读
最终回答是否指向 artifact

如果启用 LangSmith / Langfuse，可以把一次完整 run 当作验收记录。

如果不启用，也要保留本地日志、文件产物和命令输出。
