Technical Guide

09. Sub-agents 原理：多智能体什么时候值得拆

理解 DeerFlow 的子 Agent 体系：它解决上下文隔离和任务分工问题，但只有边界清楚、产出可验证时才值得使用。

多 Agent 不是越多越好

Sub-agent 更像临时同事，不是魔法。

拆得好，可以让复杂任务更清晰；拆得差，会制造更多沟通成本、上下文丢失和结果冲突。

所以学习 DeerFlow 的 Sub-agents，不要先问“能创建几个 Agent”，而要先问：

这个任务是否真的需要拆？
拆出去的子任务是否边界清楚？
子 Agent 的结果是否可验证？
Lead Agent 是否有能力汇总？

DeerFlow 里的 subagents 目录

官方当前仓库里，Sub-agent 相关实现主要在：

backend/packages/harness/deerflow/subagents/
├── builtins/
│   ├── bash_agent.py
│   └── general_purpose.py
├── config.py
├── executor.py
├── registry.py
├── status_contract.py
└── token_collector.py

从目录可以看出几个关键信息：

  • 有内置子 Agent；
  • 子 Agent 需要配置；
  • 子 Agent 需要注册；
  • 子 Agent 执行有 executor；
  • 子 Agent 状态有 contract；
  • 子 Agent token 使用需要收集和管理。

这说明 Sub-agent 不是简单“再调一次模型”。它有自己的注册、执行、状态和资源管理。

Sub-agent 解决什么问题

Sub-agent 主要解决三类问题。

1. 上下文隔离

复杂任务里，不同子问题需要不同材料。

比如写一篇 DeerFlow 源码分析：

  • 一个子任务只看 Gateway；
  • 一个子任务只看 runtime；
  • 一个子任务只看 tools；
  • 一个子任务只看 sandbox。

如果全部塞给一个 Agent，上下文会很乱。拆开后，每个子 Agent 只关注自己的范围。

2. 专业分工

不同子任务可能需要不同视角。

例如：

  • 资料搜索；
  • 源码阅读；
  • 配置核对；
  • 安全风险检查；
  • 最终报告整理。

这些可以由不同子 Agent 完成，再由 Lead Agent 汇总。

3. 并行处理

如果多个子任务相互独立，就可以并行。

例如做技术选型：

  • 子 Agent A 调研方案 1；
  • 子 Agent B 调研方案 2；
  • 子 Agent C 调研方案 3；
  • Lead Agent 统一对比。

并行能节省时间，但前提是任务边界足够清楚。

什么任务适合拆

适合拆给 Sub-agent 的任务通常满足这些条件：

  • 信息源相对独立；
  • 子任务目标清楚；
  • 每个子任务有明确产出；
  • 子任务之间依赖少；
  • 最终需要综合多个视角；
  • 单个上下文会过长；
  • 子结果可以被验证。

典型场景：

  • 大型代码库阅读；
  • 多方案技术选型；
  • 深度研究报告；
  • 多模块排错；
  • PR 代码审查；
  • 安全风险扫描；
  • 文档、代码、配置交叉核验。

什么任务不适合拆

不适合拆的任务：

  • 改一行配置；
  • 查一个确定命令；
  • 写一个短说明；
  • 需求还不清楚；
  • 子任务之间强依赖；
  • 子 Agent 输出无法验证；
  • 汇总成本比执行成本还高。

这种任务直接由 Lead Agent 做更稳。

多 Agent 不是默认更高级。很多时候，少拆反而质量更高。

子任务应该怎么写

给子 Agent 的任务要具体。

不好的写法：

你去研究一下 DeerFlow。

这太大，范围不清，结果也很难用。

更好的写法：

请只查看 DeerFlow 官方仓库中 backend/packages/harness/deerflow/subagents/ 目录。
总结它包含哪些文件、每个文件大概负责什么、Sub-agent 执行链路可能从哪里开始。
不要写教程正文，只返回结构化要点和文件路径。

好的子任务至少包含：

  • 目标；
  • 范围；
  • 输入材料；
  • 输出格式；
  • 不要做什么；
  • 验证要求。

Lead Agent 的职责更重

Sub-agent 的输出不是最终答案。

Lead Agent 必须做这些事：

  • 去重；
  • 判断冲突；
  • 过滤无关信息；
  • 补齐遗漏；
  • 统一风格；
  • 回到用户目标；
  • 形成最终结论。

如果 Lead Agent 只是把多个子 Agent 结果拼接起来，多 Agent 反而会降低质量。

真正的关键不是“拆出去”，而是“收回来”。

Sub-agent 和 Tool 的区别

Tool 是具体动作。Sub-agent 是带上下文的小任务执行者。

对比一下：

场景更像什么
读取一个文件Tool
搜索一个关键词Tool
调用一个 MCP APITool / MCP
阅读三个文件并总结架构Sub-agent
比较两个方案并输出取舍Sub-agent
写一段代码并说明风险Sub-agent

Tool 更像“手”；Sub-agent 更像“临时同事”。

Sub-agent 和 Skill 的关系

Skill 可以规定什么时候拆、怎么拆、怎么汇总。

例如一个深度研究 Skill 可以要求：

如果主题涉及多个独立方向，可以拆给 2-3 个子 Agent：
- 一个负责官方资料；
- 一个负责源码或技术实现；
- 一个负责竞品或外部资料；
最后由 Lead Agent 汇总并标注来源。

这里 Skill 是方法，Sub-agent 是执行角色。

Sub-agent 也需要限制

官方目录里可以看到：

token_collector.py
status_contract.py

这说明子 Agent 不只是“开越多越好”。系统需要知道：

  • 子 Agent 消耗了多少 token；
  • 子 Agent 当前状态是什么；
  • 子 Agent 输出是否符合约定；
  • 子 Agent 是否需要被限制数量或深度。

同时，agents/middlewares/ 里也有类似 subagent_limit_middleware.py 的中间件，这类逻辑就是为了避免子 Agent 失控。

常见失控点

Sub-agent 常见问题：

  • 子任务太大，子 Agent 自己也发散；
  • 子任务太模糊，输出无法直接用；
  • 子结果没有来源，无法验证；
  • 多个子 Agent 结论冲突；
  • Lead Agent 汇总时丢掉用户约束；
  • 子 Agent 过多，token 和时间成本过高；
  • 子 Agent 修改了不该改的东西。

所以子 Agent 最好默认只读、只分析，除非明确允许它执行修改。

在终局实战里怎么用

最后的研究型 Agent 实战，可以这样使用 Sub-agents：

Lead Agent：理解研究主题，拆任务，汇总报告
Sub-agent A：查官方文档和一手资料
Sub-agent B：查源码或技术实现
Sub-agent C：查外部文章和竞品信息
Lead Agent：合并、去重、判断可信度、生成 Markdown 报告

这比让一个 Agent 从头搜到尾更稳。

但要注意：最终报告必须由 Lead Agent 统一生成，不能把三个子结果简单拼接。

这一篇的完成标准

读完这一篇，你应该能判断：

  • 什么任务适合拆给 Sub-agent；
  • 什么任务不应该拆；
  • 子任务应该怎么写；
  • Lead Agent 为什么必须负责汇总；
  • Sub-agent 和 Tool、Skill 的区别；
  • 为什么多 Agent 需要 token、状态和数量限制。
