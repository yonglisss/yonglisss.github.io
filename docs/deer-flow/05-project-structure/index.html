05. 项目结构：从目录看懂 DeerFlow

按官方仓库当前目录重新梳理 DeerFlow：前端、Gateway、Agent harness、runtime、sandbox、MCP、skills、memory、uploads 和 tracing 分别在哪里。

为什么这一篇要看目录

上一篇先画了整体架构。这一篇把架构落到仓库目录。

读 Agent 项目最怕一上来就跳进某个函数。DeerFlow 这种工程不是一个单文件 demo，它有前端、Gateway、Agent runtime、sandbox、MCP、skills、uploads、artifacts、tracing、channels 等很多层。

先看目录，可以先回答三个问题：

  1. 用户入口在哪里？
  2. Agent 运行框架在哪里？
  3. 如果我要改一个能力，应该先看哪个目录？

下面基于官方仓库当前结构整理。

根目录先看这些

官方仓库根目录可以先简化成这样：

deer-flow/
├── backend/
├── frontend/
├── skills/
├── docs/
├── docker/
├── scripts/
├── tests/
├── contracts/
├── Makefile
├── config.example.yaml
└── extensions_config.json

先记住这几个重点：

目录 / 文件作用
backend/后端、Gateway、Agent harness、sandbox、MCP、memory、uploads 等核心逻辑
frontend/Web 前端界面
skills/官方内置或示例 skills
docs/项目文档
docker/Docker / 部署相关文件
scripts/启动、部署、维护脚本
tests/测试相关内容
contracts/前后端或接口契约相关内容
config.example.yaml主配置模板
extensions_config.json扩展配置，例如部分 MCP / extension 场景

不要一开始就把所有目录都展开。第一遍只抓主线：

frontend / channels
  ↓
backend/app/gateway
  ↓
backend/packages/harness/deerflow
  ↓
tools / skills / sandbox / memory / mcp / uploads / tracing

frontend/：用户看到的 Web 界面

frontend/ 是浏览器页面。

它主要负责：

  • 展示对话和任务状态；
  • 提交用户输入；
  • 展示 streaming 输出；
  • 展示工具调用、文件和产物；
  • 调用后端 Gateway API。

学习 DeerFlow 的第一阶段，不需要深读前端。你只要知道：前端不是 Agent 本体，它是用户入口和状态展示层。

如果以后要改页面布局、增加按钮、展示更多运行状态，再进入 frontend/。

backend/：核心在这里

DeerFlow 的核心逻辑在 backend/。

当前可以先看成三层：

backend/
├── app/
├── packages/
└── docs/

其中：

  • backend/app/ 更偏应用服务层；
  • backend/packages/ 更偏可复用的 Agent harness 包；
  • backend/docs/ 是后端相关文档。

后面源码篇主要会围绕 backend/app/gateway/ 和 backend/packages/harness/deerflow/ 展开。

backend/app/gateway/：后端统一入口

backend/app/gateway/ 是 Gateway API 所在位置。

上一篇提到，官方后端架构里 Gateway API 默认端口是 8001，Nginx 会把 /api/langgraph/* 和其他 /api/* 请求转到这里。

Gateway 不是一个简单转发层。它通常会涉及：

  • threads / runs；
  • streaming；
  • models；
  • MCP；
  • skills；
  • memory；
  • uploads；
  • artifacts；
  • thread-local cleanup。

所以如果你想追“用户点击发送之后，请求先到哪里”，第一站不是 Agent 目录，而是 backend/app/gateway/。

backend/app/channels/：IM 入口

backend/app/channels/ 对应消息渠道。

DeerFlow 支持 Web 以外的入口，比如 Telegram、Slack、飞书、企业微信、钉钉等。channels 的职责是把不同平台的消息事件转成 DeerFlow 能处理的任务，再把结果发回去。

入门阶段不建议先接 IM。因为 IM 接入会额外引入：

  • 平台 token；
  • webhook / event subscription；
  • 网络回调；
  • 消息格式转换；
  • 权限和会话映射。

先把 Web 和本地任务跑稳，再接 channels 更好排错。

backend/packages/harness/deerflow/：Agent harness 本体

这是最关键的目录。

官方仓库当前结构里，backend/packages/harness/deerflow/ 下有这些核心模块：

deerflow/
├── agents/
├── community/
├── config/
├── guardrails/
├── mcp/
├── models/
├── persistence/
├── reflection/
├── runtime/
├── sandbox/
├── skills/
├── subagents/
├── tools/
├── tracing/
├── tui/
├── uploads/
└── utils/

这就是 DeerFlow 的能力地图。

agents/：主控 Agent 和中间件

agents/ 是理解执行链路时最重要的目录之一。

它通常会涉及：

  • Lead Agent；
  • middleware chain；
  • system prompt；
  • memory context；
  • working directory guidance；
  • 工具和子 Agent 的组合。

如果你想知道“DeerFlow 为什么下一步调用这个工具”，最终要回到 agents/。

但第一次读源码不要直接跳进去。先看 runtime/、Gateway 和执行链路，再回头看 Agent，会更顺。

runtime/：运行过程的组织层

runtime/ 是这次目录梳理里要重点补上的模块。

Agent 不是孤立函数。一次任务会涉及 thread、run、streaming、中间状态、工具调用和结果返回。runtime/ 更接近“把一次运行组织起来”的层。

后面源码篇追执行链路时，会重点看它和 Gateway、Lead Agent 之间的关系。

tools/、mcp/、community/：外部能力入口

这几个目录都和“Agent 能调用什么”有关。

  • tools/：内置工具，例如文件、搜索、执行等能力；
  • mcp/：MCP server / MCP tool 接入相关逻辑；
  • community/：社区工具或第三方能力相关模块。

后面 Tools 模块会按这个顺序讲：

Tools 原理 → Tools 源码 → 自定义 Tool 实战

MCP 会单独拆，因为它解决的是外部工具协议接入，不只是一个内置函数。

skills/：可复用工作方法

仓库根目录有 skills/，harness 包里也有 deerflow/skills/。

可以先这样区分：

  • 根目录 skills/ 更像可加载的 skill 内容；
  • deerflow/skills/ 更像 skill 加载、解析、注入的代码实现。

Skill 不是工具调用本身。它更像一份可复用的工作方法，会影响 Agent 怎么理解和执行某类任务。

所以后面 Skills 模块也会拆成三篇：

Skills 原理 → Skills 源码 → 写一个 Skill

sandbox/：执行边界

sandbox/ 和文件、命令、代码执行密切相关。

只要 Agent 能执行命令、写文件、跑代码，就必须考虑边界。Sandbox 负责把这些执行放到更可控的环境中。

读这个目录时，不要只看“怎么启动容器”。更应该关注：

  • 工作目录怎么创建；
  • 文件如何挂载；
  • 工具在 sandbox 内怎么执行；
  • 输出怎么回到 artifacts；
  • 多个 thread / run 怎么隔离。

persistence/、uploads/、tracing/

这几个目录决定 DeerFlow 是否像一个真实系统，而不是一次性脚本。

目录关注点
persistence/状态、数据、持久化相关逻辑
uploads/用户上传文件如何进入任务上下文
tracing/LangSmith / Langfuse 等可观测性接入

Agent 项目跑 demo 很容易，长期可用难在这些地方。

如果任务失败后你看不到中间过程，就很难排错；如果上传、产物和状态没有组织好，用户也很难复用结果。

models/、config/、guardrails/

这几个目录影响 Agent 的运行质量和边界。

  • models/：不同模型 provider 的适配；
  • config/：配置加载、解析和默认值；
  • guardrails/：运行边界、约束或安全相关能力。

模型问题不只是 API key。不同 provider 对 tool call、vision、thinking、streaming 的支持都可能不同，所以需要模型适配层。

配置问题也不只是 YAML 语法。很多行为都是配置驱动的：模型、搜索、sandbox、skills、MCP、tracing、token budget 都会受配置影响。

建议读源码的顺序

不要按目录字母顺序读。

更建议这样读：

1. backend/app/gateway/        看请求入口
2. deerflow/runtime/           看一次 run 怎么组织
3. deerflow/agents/            看 Lead Agent 和 middleware
4. deerflow/tools/             看工具怎么注册和调用
5. deerflow/sandbox/           看执行边界
6. deerflow/skills/            看 skill 怎么加载和注入
7. deerflow/subagents/         看子 Agent 怎么创建和汇总
8. deerflow/mcp/               看外部工具怎么接入
9. deerflow/persistence/       看状态和数据怎么保存
10. deerflow/tracing/          看执行过程怎么观测

这条路线和任务执行链路一致，比从根目录随机点开更稳。

这一篇的完成标准

读完这一篇，你应该能做到：

  • 知道 Web 页面在 frontend/，不是 Agent 本体；
  • 知道 Gateway 在 backend/app/gateway/；
  • 知道 Agent harness 在 backend/packages/harness/deerflow/；
  • 能区分 tools/、skills/、subagents/、sandbox/、mcp/ 的职责；
  • 知道后面读源码应该沿着请求链路读，而不是随机读目录。
