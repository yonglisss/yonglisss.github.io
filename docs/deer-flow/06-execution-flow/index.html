Technical Guide

06. 执行链路：一次任务从入口到结果

沿着真实运行路径理解 DeerFlow：入口、Gateway、runtime、Lead Agent、middleware、tools、sandbox、artifacts 和 streaming 如何协作。

这一篇追一条主线

前面两篇分别做了两件事：

  • 04 画整体架构图；
  • 05 把架构落到仓库目录。

这一篇继续往前走：不再列模块，而是追一条任务链路。

先不用纠结每个函数名。你只要抓住一件事：DeerFlow 不是“前端把消息丢给模型，然后模型回答”。它中间有 Gateway、runtime、middleware、Lead Agent、tools、sandbox、artifacts 和 streaming。

简化链路

可以先记住这个版本：

用户提交任务
  ↓
Frontend / API / IM Channel
  ↓
Nginx / Gateway API
  ↓
创建 thread / run
  ↓
Runtime 组织执行
  ↓
Middleware 准备上下文和工作环境
  ↓
Lead Agent 选择模型、工具和下一步动作
  ↓
Tools / MCP / Sub-agents / Sandbox 执行
  ↓
Artifacts / Memory / Persistence 记录结果
  ↓
Streaming 返回过程和最终结果

后面源码篇会把这条链路拆到具体文件。现在先建立运行感觉。

1. 用户入口不止 Web

用户任务可以从多个入口进来：

  • Web 前端；
  • LangGraph-compatible API；
  • 普通 Gateway API；
  • Telegram、Slack、飞书、企业微信、钉钉等 IM channels；
  • Python client 或其他嵌入调用。

入门阶段通常从 Web 页面开始，因为它最直观。但从系统角度看，Web 只是入口之一。

这也是为什么 DeerFlow 有 Gateway：不同入口最终都要转换成统一的后端任务执行。

2. Nginx 和 Gateway 先接住请求

官方后端架构里，Nginx 位于最外层，常见路径会被路由到 Gateway：

/api/langgraph/*  → LangGraph-compatible API
/api/*            → Gateway API 其他接口
/                 → Frontend

Gateway 的职责不只是转发。它还要处理：

  • thread；
  • run；
  • streaming；
  • models；
  • uploads；
  • artifacts；
  • memory；
  • skills；
  • MCP；
  • thread-local cleanup。

所以如果页面上点击发送后没反应，排查不要直接跳到模型。更合理的顺序是：前端请求是否发出、Gateway 是否收到、run 是否创建、runtime 是否开始执行。

3. Thread 和 Run 是任务容器

Agent 系统里，一次任务不能只看成一条消息。

更合理的理解是：

thread：一段会话或任务上下文
run：thread 里的某次执行

为什么需要这层？因为一次长任务可能包括：

  • 用户上传的文件；
  • 多轮消息；
  • 工具调用结果；
  • 中间输出；
  • 最终 artifact；
  • 可恢复或可观察的运行状态。

如果没有 thread / run 这样的抽象，所有状态都会混在聊天记录里，很难管理。

4. Runtime 负责把任务跑起来

Gateway 接到请求后，真正跑任务的是 runtime 层。

你可以把 runtime 理解成“执行调度层”。它不一定亲自做所有推理，但会组织一次 run：

  • 初始化运行上下文；
  • 准备事件流；
  • 调用 Lead Agent；
  • 接收工具调用和中间事件；
  • 把结果桥接回 Gateway；
  • 必要时处理取消、清理和异常。

后面源码篇会重点看 deerflow/runtime/ 与 Gateway、Agent 之间的关系。

5. Middleware 先整理运行环境

在 Lead Agent 真正工作前后，middleware 会处理很多横切逻辑。

这一步很容易被忽略，但它决定了 Agent 能不能稳定执行。

典型职责包括：

  • 创建每个 thread 的工作目录；
  • 处理用户上传文件；
  • 注入 memory；
  • 准备 system prompt；
  • 注入 skills；
  • 限制 token budget；
  • 处理 artifacts；
  • 维护工作目录说明；
  • 清理运行状态。

如果你发现 Agent “看不到上传文件”“写错目录”“没有使用 skill”“上下文丢了”，问题经常不在模型，而在 middleware 准备的上下文和环境。

6. Lead Agent 开始判断下一步

Lead Agent 是主控。

它拿到整理后的上下文后，会判断：

  • 用户到底要什么结果；
  • 当前信息够不够；
  • 是否需要搜索；
  • 是否需要读写文件；
  • 是否需要执行命令；
  • 是否需要调用 MCP 工具；
  • 是否需要拆给子 Agent；
  • 是否应该直接给最终答案。

普通聊天模型通常直接回答。Agent 不一样，它更像一个决策器：先决定下一步动作，再根据动作结果继续推理。

7. 模型选择和工具选择发生在一起

DeerFlow 不是只固定调用一个模型。

模型选择可能受配置影响，也可能跟任务能力有关，比如：

  • 是否需要 vision；
  • 是否需要 thinking / reasoning；
  • 是否支持 tool call；
  • 是否适合快速回复；
  • 是否适合长任务推理。

同时，模型会在当前可用工具中选择下一步动作。

这也是为什么配置很重要：如果模型 provider 不支持工具调用，或者工具没有正确注册，Agent 的能力会明显下降。

8. Tools / MCP 真正执行动作

当 Lead Agent 判断需要外部能力时，会调用工具。

常见工具类型包括：

  • 搜索；
  • 网页抓取；
  • 文件读取；
  • 文件写入；
  • bash / Python 执行；
  • 图片或上传文件处理；
  • MCP server 暴露的外部工具。

这里要记住一句话：

工具结果不是最终答案，而是下一轮推理的输入。

比如搜索返回 5 个链接，Agent 还要判断哪些来源可信、是否需要抓全文、是否和用户任务相关。

9. Sandbox 承接高风险动作

只要任务涉及代码执行、命令执行、文件读写，就要考虑 sandbox。

Sandbox 的作用不是让 Agent 更聪明，而是让执行更可控。

它会影响：

  • 命令在哪里执行；
  • 文件写到哪里；
  • 宿主机目录如何挂载；
  • 多个 thread 是否隔离；
  • 产物如何从执行环境回到系统；
  • 失败时如何清理。

所以当你排查“命令为什么没跑成功”“文件为什么找不到”“生成结果在哪里”时，要把 sandbox 纳入链路，而不是只看工具定义。

10. Sub-agents 处理可拆分任务

复杂任务可能会拆给 sub-agents。

适合拆的情况：

  • 多个资料源可以并行查；
  • 不同子问题需要不同视角；
  • 任务很长，需要隔离上下文；
  • 需要一个 Agent 做检查或复核。

不适合拆的情况：

  • 任务很短；
  • 步骤强依赖，不能并行；
  • 子任务描述不清；
  • 拆分成本大于收益。

多 Agent 的价值是分工，不是数量。后面 Sub-agents 模块会单独讲原理、源码和一个小实战。

11. Artifacts 把结果变成产物

DeerFlow 更像工作台，不是只在聊天框里回一句话。

一次任务可能生成：

  • Markdown 报告；
  • 代码文件；
  • 配置文件；
  • 图表；
  • 分析中间结果；
  • 下载或处理后的文件。

这些产物会通过 artifacts / uploads / persistence 相关机制被管理，并最终展示给用户。

如果你要做“AI 帮我完成任务”，这一步很关键。真正有复用价值的是文件和产物，而不只是聊天窗口中的解释。

12. Streaming 让长任务可观察

Agent 长任务如果只等最后结果，体验会很差，也很难排错。

Streaming 的价值是把过程暴露出来：

  • 当前正在思考；
  • 正在调用什么工具；
  • 工具返回了什么；
  • 是否生成了文件；
  • 是否遇到错误；
  • 最终结果是什么。

如果任务卡住，要判断它卡在哪一层：

Frontend 没收到事件？
Gateway 没创建 run？
Runtime 没启动？
Lead Agent 没继续？
Tool / sandbox 卡住？
Streaming 没转发？

这比笼统地说“Agent 挂了”更有用。

一次任务的排错顺序

遇到问题时，可以按这条顺序查：

1. 前端请求是否发出
2. Gateway 是否收到请求
3. thread / run 是否创建
4. runtime 是否开始执行
5. middleware 是否准备好上下文
6. Lead Agent 是否产生下一步动作
7. tool / MCP / sandbox 是否执行成功
8. artifacts 是否生成
9. streaming 是否返回
10. 前端是否正确展示

这条顺序后面会反复用到。

这一篇的完成标准

读完这一篇，你应该能用一条线解释 DeerFlow 的执行过程：

入口 → Gateway → thread/run → runtime → middleware → Lead Agent → tools/sandbox/sub-agents → artifacts → streaming

下一步开始进入能力模块。建议先从 Tools 开始，因为 Tools 是 Agent 从“会说”变成“会做”的第一层能力。
