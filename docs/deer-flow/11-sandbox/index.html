Technical Guide

11. Sandbox：安全执行和文件系统

理解 sandbox 和文件系统边界：为什么需要隔离、能隔离什么、不能隔离什么，以及本地和 Docker 模式取舍。

只要能执行，就必须谈边界

Agent 一旦能执行命令、写文件、运行代码，就不再只是聊天工具。它开始接触真实环境。

这时最重要的问题不是“能不能做”，而是“在哪里做、能做多大范围、出错后影响什么”。Sandbox 就是为了解决这个问题。

Sandbox 解决什么问题

Sandbox 的作用是隔离执行环境。

它不能让所有风险消失，但可以降低误改宿主机文件、依赖安装污染环境、脚本副作用、多任务互相影响等风险。

你可以把 sandbox 理解成 Agent 的工作间。它可以在里面读写、执行、生成产物，但边界要明确。

DeerFlow 支持哪些执行模式

官方文档里提到的 sandbox 思路包括本地执行、Docker / Container sandbox、Docker + Kubernetes provisioner。

模式适合场景风险 / 成本
本地执行快速开发、低风险任务容易影响宿主机
Docker sandbox本地体验、常规执行任务首次拉镜像慢，占资源
Provisioner / Kubernetes共享环境、长期服务、团队使用配置复杂，运维成本高

入门阶段推荐先用 Docker 路线。

Sandbox 和 File System 的关系

Agent 可能需要读取输入文件、修改项目代码、保存 Markdown 报告、生成图表、写临时脚本。

这些文件写在哪里，取决于运行目录、挂载目录和 sandbox 配置。

如果 Agent 说文件已生成但页面找不到产物，可能是写到 sandbox 临时目录、写到错误工作目录、没有被 artifact 系统收集、权限不足或前端没刷新。

Docker 模式为什么第一次慢

Docker 模式第一次启动常见慢点：拉取 sandbox 镜像、构建镜像、安装依赖、挂载源码、启动多个服务。

可以先执行：

make setup-sandbox

或 Docker 开发模式：

make docker-init
make docker-start

第一次慢是正常的，第二次通常会快很多。

本地执行的风险

本地执行最方便，也最危险。

建议在干净工作区测试，不给高权限用户运行，修改前先看 diff，写文件后立即验证，对危险命令保持人工确认。

Sandbox 不是绝对安全

仍然要考虑容器挂载、网络权限、环境变量密钥、文件系统权限、Docker socket 是否暴露、任务是否来自不可信输入。

如果把 DeerFlow 暴露到公网或团队环境，必须额外做认证、权限、审计和资源限制。

文件写入怎么控制

至少控制路径、覆盖、验证三件事。

代码项目写完后看 diff 并运行测试；文档项目写完后跑 build；数据文件检查格式和行数。

完成标准

读完后，你应该知道 sandbox 为什么是执行型 Agent 的核心边界，也知道它不能替代权限、认证和审计。
