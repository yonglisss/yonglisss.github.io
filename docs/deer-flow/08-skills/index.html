08. Skills：把一次经验沉淀成可复用能力

讲清 Skill 的定位、适用场景、写作原则和最小可用 Skill 的设计方式。

Skill 不是更长的 Prompt

Prompt 更像临时指令。Skill 更像可复用的工作方法。

如果你经常让 Agent 做同一类事，比如写技术教程、排查部署问题、生成研究报告、修改某类配置，就适合沉淀成 skill。

Skill 解决什么问题

Agent 失败有时不是模型不够强，而是每次都从零开始。

例如写开发者教程时，它需要记住：先查官方文档，不要凭旧博客脑补，给出可复制命令，写完要跑 build，只提交目标文件，语气要克制。

这些要求每次都写在 prompt 里，很容易漏。Skill 的作用，就是把稳定流程沉淀下来。

Skill 和 Tool 的区别

类型解决的问题示例
Prompt这一次怎么做这篇写短一点
Skill同类任务以后怎么做写教程时先查官方资料再构建验证
Tool具体执行动作读取文件、执行命令、搜索网页
MCP外部系统工具化GitHub、数据库、文档系统

Skill 本身通常不直接执行动作。它指导 Agent 选择什么步骤、调用哪些工具、怎么验证结果。

什么适合做成 Skill

适合沉淀成 skill 的任务通常有几个特征：重复出现、步骤固定、容易踩坑、需要验证、能节省沟通。

比如：写开发者教程、发布静态站点、排查 GitHub Pages 缓存、同步知识库、修复某类项目构建问题。

什么不适合做成 Skill

一次性资料、很快过期的数据、临时任务进度、某次提交 hash、某个 PR 编号、没验证过的猜测，都不适合。

Skill 是长期能力，不是垃圾桶。错误或过期 skill 会让 Agent 反复犯错。

一个好 Skill 应该包含什么

至少包含：什么时候使用、目标是什么、标准流程、常见坑、验证方式、必要命令或模板。

坏 skill 往往是“写文章要有深度、逻辑清楚”。这没有实际帮助。

更好的写法是：写开发者教程时，先读取官方 README 和配置示例；正文按“问题 → 步骤 → 验证 → 常见坑”组织；写完后运行真实 build；官方资料拿不到就不要脑补。

Skill 如何参与一次任务

当 Agent 判断任务和某个 skill 相关时，它会把 skill 中的流程和约束带入当前任务。

例如“继续写 DeerFlow 教程，并提交推送”，skill 可以提醒它：先查当前文件，继续已有风格，不切换成营销表达，写完跑 check/build，只提交目标文件。

Skill 也需要维护

项目命令、目录、用户偏好、官方文档都会变化。Skill 不维护，就会从经验变成误导。

和 Memory 的区别

Memory 保存稳定事实，Skill 保存可复用流程。

不要把流程写进 Memory，也不要把临时事实写进 Skill。

完成标准

读完后，你应该能分清 Skill、Prompt、Tool、Memory 的边界，并知道什么任务值得沉淀成 Skill。
