Technical Guide

11. Sandbox 原理：安全执行和文件边界

理解 DeerFlow 的 sandbox：为什么执行必须隔离，LocalSandboxProvider 和 Docker sandbox 的边界在哪里，文件产物如何被安全管理。

只要 Agent 能执行，就必须谈边界

Agent 一旦能执行命令、读写文件、运行代码，就不再只是聊天工具。

它可能会：

  • 安装依赖；
  • 运行 bash；
  • 读取项目文件；
  • 生成报告；
  • 修改源码；
  • 处理上传文件；
  • 调用外部服务。

这些能力很有用，但也很危险。

Sandbox 解决的就是“让执行发生在更可控边界内”的问题。

DeerFlow 的 sandbox 相关目录

官方当前仓库里，sandbox 相关代码主要在：

backend/packages/harness/deerflow/sandbox/
├── exceptions.py
├── file_operation_lock.py
├── local/
│   ├── list_dir.py
│   ├── local_sandbox.py
│   └── local_sandbox_provider.py
├── middleware.py
├── sandbox.py
├── sandbox_provider.py
├── search.py
├── security.py
└── tools.py

社区 Docker / AIO sandbox 相关实现主要在：

backend/packages/harness/deerflow/community/aio_sandbox/

配置模板里默认可以看到：

sandbox:
  use: deerflow.sandbox.local:LocalSandboxProvider
  allow_host_bash: false

官方注释也很明确：LocalSandboxProvider 默认禁用 host bash，因为它不是安全隔离边界。只有完全可信的单用户本地工作流才应该打开。

Sandbox 解决什么问题

Sandbox 主要解决四件事。

1. 执行位置

命令和文件操作不能随便在宿主机任意位置执行。

系统需要知道：

  • 当前工作目录在哪里；
  • 用户上传文件在哪里；
  • 输出文件应该保存在哪里；
  • 哪些路径可以访问；
  • 哪些路径不该访问。

2. 文件边界

Agent 很容易写错路径。

比如它以为自己在项目目录，实际在临时目录；或者它生成了报告，但前端找不到；或者多个任务同时写同一个文件。

Sandbox 需要管理虚拟路径、挂载和文件操作锁。

3. 命令风险

Bash 很强，但风险也最大。

它可能：

  • 删除文件；
  • 暴露密钥；
  • 安装不可信依赖；
  • 长时间占用资源；
  • 修改宿主环境；
  • 访问不该访问的目录。

所以 host bash 默认关闭是合理的。

4. 多任务隔离

如果多个 thread / run 同时运行，它们不能互相污染。

否则一个任务的上传文件、输出报告、临时目录可能被另一个任务误读或覆盖。

Local sandbox 和 Docker sandbox 的区别

可以先这样理解：

模式优点风险 / 成本
LocalSandboxProvider简单、启动快、适合本地可信环境不是强隔离，host bash 默认禁用
Docker / AIO sandbox隔离更强，适合执行代码和长期任务启动慢，依赖 Docker，排错层更多

入门阶段可以先用 local 跑通，但只要涉及真实命令执行、团队使用、处理不可信输入，就应该认真考虑 Docker sandbox 或更强隔离。

Sandbox 不是绝对安全

Sandbox 能降低风险，但不是万能保险。

你仍然需要：

  • 限制可访问目录；
  • 控制是否允许 bash；
  • 不把密钥暴露给 Agent；
  • 检查写文件路径；
  • 对危险命令做 guardrails；
  • 记录执行过程；
  • 对生成文件做验证。

不要把“启用了 sandbox”等同于“可以让 Agent 随便执行”。

文件操作为什么需要锁

官方目录里有：

file_operation_lock.py

这类机制说明文件操作不是小事。

并发任务中，可能出现：

  • 两个工具同时写同一文件；
  • 一个任务读文件时另一个任务正在改；
  • 生成产物还没完成就被展示；
  • 临时文件被提前清理。

文件锁和工作目录边界，是 Agent 可靠交付文件的基础。

Sandbox 和 Tools 的关系

很多工具最终会落到 sandbox。

例如：

read_file
write_file
str_replace
bash
ls

这些工具看起来只是“文件/命令工具”，但真正重要的是：它们在哪里执行、能看到哪些文件、有没有权限、输出如何返回。

所以 Tools 模块不能脱离 Sandbox 看。

Sandbox 和 Artifacts 的关系

任务产物通常先在执行环境里生成，再通过 artifacts 机制展示给用户。

如果你发现“文件生成了但页面看不到”，可能的问题包括：

  • 文件写在错误目录；
  • sandbox 路径没有映射到 artifacts；
  • 输出没有被 present / register；
  • 文件生成失败但 Agent 误以为成功；
  • 前端没有展示对应 artifact。

排查时要沿着路径走：

工具调用 → sandbox 文件路径 → artifacts 注册 → Gateway 返回 → 前端展示

终局实战里怎么用 Sandbox

研究型 Agent 最终会生成 Markdown 报告。

推荐边界是：

  • 搜索和抓取可以走工具；
  • 报告写入固定输出目录；
  • 写完后读取文件验证；
  • 不允许随意写项目根目录；
  • 不执行不必要 bash；
  • 如果要跑脚本，放进 sandbox；
  • 最终报告通过 artifacts 展示。

这样用户拿到的是文件产物，而不是聊天框里一大段不可复用文本。

常见排错点

Sandbox 常见问题：

  • allow_host_bash 没开，bash 工具不可用；
  • Docker 没启动或镜像没准备好；
  • 挂载路径在容器内不可见；
  • 文件写到了 sandbox 内部，但没有回到 artifacts；
  • 多任务并发写同一路径；
  • Agent 误把虚拟路径当宿主机路径；
  • 权限不足导致读写失败。

排错时先确认三件事：

当前使用哪个 sandbox provider？
工具实际在哪个工作目录执行？
文件产物是否被注册给前端？
