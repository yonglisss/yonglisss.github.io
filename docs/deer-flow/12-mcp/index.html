Technical Guide

12. MCP 原理：把外部工具接进 Agent

理解 DeerFlow 的 MCP 接入：client、session pool、OAuth、tool cache 和 tools 转换如何让外部系统成为 Agent 可调用能力。

MCP 解决的是外部工具接入问题

内置工具再多，也不可能覆盖所有系统。

你可能希望 Agent 调用：

  • GitHub；
  • 数据库；
  • 内部知识库；
  • 浏览器自动化；
  • 文档系统；
  • 工单系统；
  • 企业 API；
  • 云服务控制台。

如果每接一个系统都写一套私有工具协议，维护成本会很高。

MCP 的价值就是把外部工具以统一协议暴露给 Agent。

DeerFlow 里的 MCP 目录

官方当前仓库里，MCP 相关代码主要在：

backend/packages/harness/deerflow/mcp/
├── cache.py
├── client.py
├── oauth.py
├── session_pool.py
└── tools.py

Gateway 里也有 MCP 路由：

backend/app/gateway/routers/mcp.py

后端 README 里列了相关 API：

GET/PUT /api/mcp/config
POST /api/mcp/cache/reset

这些信息说明 MCP 在 DeerFlow 里不是一个简单工具函数，而是一套配置、连接、缓存和工具转换机制。

MCP 接入链路

可以先简化成这样：

MCP server 配置
  ↓
Gateway / config 加载
  ↓
MCP client 建立连接
  ↓
session pool 维护会话
  ↓
拉取 server 暴露的 tools
  ↓
cache 缓存工具信息
  ↓
转换成 Agent 可用 tool
  ↓
Lead Agent 调用

这个链路比普通内置工具长，所以排错时也要分层看。

为什么需要 session pool

MCP server 可能不只是一次 HTTP 请求。

它可能需要维持连接、初始化能力、复用 session、处理多个 tool 调用。

session_pool.py 的存在说明 DeerFlow 需要管理 MCP 会话，而不是每次工具调用都从零创建连接。

这会影响：

  • 性能；
  • 连接稳定性；
  • 多 server 管理；
  • 并发调用；
  • 失败恢复。

为什么需要 OAuth

oauth.py 说明 MCP 接入可能涉及授权。

很多外部服务不是只靠一个本地函数就能调用，它们需要：

  • token；
  • OAuth 流程；
  • 权限范围；
  • 用户身份；
  • 过期刷新；
  • 安全存储。

所以 MCP 工具接入时，不能只问“工具有没有出现”，还要问“权限是否正确”。

为什么需要 tool cache

DeerFlow 有 MCP cache，也有 Gateway API 可以 reset cached MCP tools。

缓存的好处是减少重复加载，提高性能。

但缓存也会带来排错问题：

  • MCP server 新增了工具，但 Agent 看不到；
  • 工具 schema 改了，但缓存仍是旧的；
  • server 配置更新后，没有重新加载；
  • 旧工具还在列表里，但实际不可用。

这时就需要重置 MCP cache，再重新加载工具。

MCP 和 Tool 的区别

MCP 最终会变成 Agent 可调用的工具，但它和内置 Tool 不是一回事。

类型来源典型问题
内置 ToolDeerFlow 自己实现工具定义、权限、sandbox、返回值
Community ToolDeerFlow provider 封装provider key、API 限制、网络错误
MCP Tool外部 MCP server 暴露server 配置、连接、OAuth、schema、缓存

所以 MCP 出问题时，不要只看 Agent。要先确认 MCP server 本身可用。

配置层要先跑通

MCP 接入前，建议先确认：

1. MCP server 能独立启动
2. 配置文件路径正确
3. Gateway 能读取 MCP config
4. MCP tools 能被列出来
5. Agent 能看到工具名
6. 工具调用返回结构符合预期

不要一开始就写复杂任务。先让一个最小工具调用成功。

Deferred Tool Loading 和 tool_search

config.example.yaml 里提到一个重要机制：当 tool_search 开启时，MCP tools 不会全部直接塞进 Agent 上下文，而是通过工具名列表和 tool_search 在运行时发现。

这样做的原因是：MCP server 多了以后，工具数量可能很大。

如果把所有工具 schema 都塞进 prompt，会带来：

  • 上下文变长；
  • 模型选择困难；
  • 成本上升；
  • 相关工具反而被淹没。

Deferred Tool Loading 的思路是：先让 Agent 知道有哪些工具，再按需检索详细工具。

这对大规模 MCP 接入很重要。

MCP 的安全边界

MCP 能接外部系统，所以风险也更高。

需要关注：

  • MCP server 是否可信；
  • token 权限是否过大；
  • 工具是否可能修改外部数据；
  • 工具 schema 是否清楚；
  • 是否需要人工确认；
  • 日志里是否会暴露密钥；
  • 返回值是否可能过大。

尤其是能写数据、发消息、删文件、改 issue、操作数据库的 MCP 工具，应该有更严格的权限和确认机制。

终局实战里怎么用 MCP

研究型 Agent 不一定必须接 MCP，但接一个简单 MCP 会让实战更完整。

推荐先接低风险工具，比如：

  • 只读知识库搜索；
  • GitHub 仓库只读查询；
  • 文档系统只读搜索；
  • 本地文件索引查询。

不要一开始就接“能写数据库 / 发消息 / 修改生产系统”的 MCP。

终局实战可以这样设计：

研究主题输入
  ↓
搜索公开资料
  ↓
调用 MCP 查询内部知识库或 GitHub 仓库
  ↓
合并来源
  ↓
生成 Markdown 报告

这样 MCP 是增强资料来源，而不是让实战变成权限排错。

常见排错顺序

MCP 出问题时，可以按这个顺序查：

1. MCP server 是否能独立启动
2. 配置是否被 Gateway 读取
3. OAuth / token 是否有效
4. tools 是否被加载
5. cache 是否需要 reset
6. Agent 是否能看到工具
7. tool schema 是否清楚
8. 调用结果是否结构化、不过大
9. Agent 是否正确使用结果

不要直接归因于模型。
