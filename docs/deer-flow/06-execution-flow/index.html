Technical Guide

06. 一次任务的完整执行链路

从用户输入到最终结果，拆解 DeerFlow 2.0 中 Gateway、Agent Runtime、模型、工具、沙盒、子 Agent 和文件产物的协作过程。

为什么要看执行链路

前面两篇分别看了核心概念和项目结构。

但只知道目录还不够。真正理解一个 Agent 系统，要看一次任务是怎么流动的：用户输入从哪里进来，谁负责调度，工具什么时候被调用，文件怎么生成，结果怎么回到页面。

这一篇不追源码细节，只建立主线。你后面读代码、排错、做二开，都可以拿这条链路当地图。

先看一张简化图

可以先把 DeerFlow 的执行链路简化成这样：

用户输入
  ↓
Frontend / API / IM Channel
  ↓
Gateway API
  ↓
Agent Runtime
  ↓
Lead Agent
  ↓
模型推理 + 工具选择
  ↓
Tools / Skills / Sub-agents / Sandbox / Memory
  ↓
文件、状态、工具结果回写
  ↓
Gateway 流式返回
  ↓
Frontend 展示最终结果和产物

真实实现会更复杂，但这条线足够帮助你定位问题。

1. 用户输入从哪里来

DeerFlow 的用户入口不止一个。

常见入口包括：

  • Web 前端；
  • API；
  • Telegram、Slack、飞书、企业微信、钉钉等 IM 渠道；
  • Python Client 或其他嵌入式调用方式。

入门阶段最常用的是 Web 前端。

你在页面里输入一段任务，比如：

帮我研究 DeerFlow 2.0 的核心架构，并生成一份 Markdown 报告。

前端不会自己执行任务，它会把输入交给后端 Gateway。

2. Gateway 接住请求

Gateway 是后端服务入口。

官方说明里提到，DeerFlow 的 Gateway API 默认运行在 8001，而完整应用会通过 Nginx 提供统一入口，例如：

http://localhost:2026

Nginx 会把相关 API 请求转到 Gateway。

Gateway 这层大致负责：

  • 接收前端或渠道入口的请求；
  • 管理 thread / run；
  • 处理上传文件、产物、memory、skills、MCP 等接口；
  • 把真正的任务交给 Agent runtime；
  • 把运行过程和最终结果返回给调用方。

所以如果前端点了发送但没有任何反应，排查顺序通常是：前端请求、Gateway 日志、Agent runtime 日志。

3. Agent Runtime 开始执行

DeerFlow 2.0 的运行时不是简单调一个模型接口。

官方架构说明里提到，Agent runtime 会通过类似 RunManager、run_agent()、StreamBridge 这类组件运行，并通过 Gateway 对外暴露 LangGraph-compatible API。

这里不需要一开始记住所有类名，只要理解：

  • Gateway 负责接请求；
  • Agent Runtime 负责跑任务；
  • Runtime 运行过程中会不断产生事件；
  • 这些事件会被桥接成前端能看到的流式输出。

这也是为什么 DeerFlow 可以展示任务过程，而不只是等最后一句结果。

4. Lead Agent 判断下一步

进入 Agent Runtime 后，任务会交给 Lead Agent。

Lead Agent 首先要做判断：

  • 用户到底要什么结果；
  • 任务是否需要联网搜索；
  • 是否需要读写文件；
  • 是否需要执行命令；
  • 是否需要调用子 Agent；
  • 是否需要使用已有 skill；
  • 当前上下文是否足够。

这一步很关键。很多 Agent 失败，第一步就已经走错了。

比如用户只是问“DeerFlow 是什么”，Lead Agent 不一定需要调用工具。但如果用户问“根据官方 README 总结 DeerFlow 2.0 的安装步骤”，那就应该优先读取官方资料，而不是凭记忆回答。

5. 模型不是直接给最终答案

在普通聊天里，模型通常直接生成答案。

但在 Agent 系统里，模型更像决策器。它可能会先输出一个工具调用意图，比如：

  • 搜索官方文档；
  • 读取某个文件；
  • 创建一个报告文件；
  • 调用 bash 检查环境；
  • 分配一个子 Agent 做资料整理。

这就是 Agent 和普通聊天的重要区别：模型不只是“说”，还会“选择动作”。

当然，动作是否真的能执行，取决于系统开放了哪些工具、配置了哪些权限，以及 sandbox 是否可用。

6. Tools 被调用

当 Lead Agent 判断需要外部能力时，就会调用 tools。

常见 tool 类型包括：

  • web search；
  • web fetch / crawl；
  • 文件读取；
  • 文件写入；
  • bash 执行；
  • 图片查看；
  • MCP server 暴露的工具。

工具调用通常会返回结构化结果。模型再根据这些结果继续推理。

这里有一个重要原则：

工具调用结果不是最终答案，它只是下一步推理的输入。

比如搜索工具返回了 5 个网页，Agent 还要判断哪些来源可信、哪些内容和任务有关、是否需要继续抓取全文。

7. Skills 影响执行方式

如果任务命中了某个 skill，Agent 的执行方式会被 skill 影响。

Skill 不一定直接执行动作，它更像一套工作方法。例如：

  • 做开发者教程时，要先确认官方文档；
  • 修改代码后，要跑真实 build；
  • 写报告时，要先确认来源完整性；
  • 做部署时，要先检查配置和日志。

所以 skill 的作用不是替代 tool，而是指导 Agent 如何组织工具和步骤。

这也是 DeerFlow 值得关注的点：它不是只靠一次 prompt，而是可以把稳定经验变成系统能力。

8. Sub-agents 处理子任务

当任务比较复杂时，Lead Agent 可能会把一部分工作交给子 Agent。

比如一个研究任务可以拆成：

  • 子 Agent A：查官方文档；
  • 子 Agent B：查源码结构；
  • 子 Agent C：整理竞品对比；
  • Lead Agent：汇总结果并写报告。

子 Agent 的价值在于隔离上下文和分工。但它也有成本：

  • 子任务描述要清楚；
  • 子 Agent 可能理解偏；
  • 汇总时可能丢细节；
  • 并行执行会增加资源消耗。

所以不要迷信多 Agent。任务足够复杂时再拆，才有意义。

9. Sandbox 承接高风险执行

如果任务涉及命令执行、代码运行、文件写入，sandbox 就会参与。

它的作用是给执行能力加边界。

举几个例子：

  • 让 Agent 跑一个 Python 脚本；
  • 生成一个 Markdown 文件；
  • 读取某个目录下的项目文件；
  • 执行 bash 命令检查依赖；
  • 修改代码并运行测试。

这些动作如果直接在宿主机上裸跑，风险比较高。sandbox 可以把执行范围控制在更可管理的环境里。

但要注意：sandbox 不是万能保险。你仍然需要限制权限、控制目录、检查输出，并避免让 Agent 执行不可信命令。

10. Memory 提供长期上下文

Memory 会在合适的时候参与任务。

它可能提供：

  • 用户偏好；
  • 项目固定路径；
  • 常用模型或工具约定；
  • 历史沉淀的工作方法。

但 memory 不应该替代实时检查。

比如 memory 里写着“项目使用某个路径”，但你要改文件时，仍然应该实际检查仓库当前状态。因为项目可能已经变了。

长期记忆提供方向，当前工具结果提供事实。这两个要分清。

11. 文件和产物被保存

DeerFlow 更像工作台，而不是单纯聊天框，一个重要原因就是它可以生成产物。

产物可能是：

  • Markdown 报告；
  • 代码文件；
  • 配置文件；
  • 图片或图表；
  • 中间分析结果；
  • 下载或上传的文件。

这些产物一般会通过文件系统和 artifacts 相关接口管理，并在前端展示。

如果你要做“AI 帮我完成任务”，而不是“AI 给我建议”，这一步很关键。真正可复用的是产物，而不是聊天窗口里的一段回答。

12. 结果流式返回

任务执行过程中，Gateway 会把运行事件返回给前端。

前端可能展示：

  • 当前正在思考或执行；
  • 工具调用过程；
  • 中间状态；
  • 最终回答；
  • 生成的文件或 artifact。

这类流式返回对长任务很重要。否则用户只能等一个黑盒结果，不知道卡在搜索、模型、sandbox 还是文件写入。

如果你在使用时发现任务“卡住”，要先判断：

  • 是前端没有收到事件；
  • 还是 Gateway 没有继续输出；
  • 还是 Agent 在等待工具；
  • 还是模型请求超时；
  • 还是 sandbox 执行慢。

一个具体例子

假设你给 DeerFlow 一个任务：

请基于官方资料整理 DeerFlow 2.0 的快速开始教程，并保存为 Markdown 文件。

理想执行链路大概是：

  1. 前端把任务发给 Gateway。
  2. Gateway 创建或继续一个 thread / run。
  3. Agent Runtime 启动 Lead Agent。
  4. Lead Agent 判断需要官方资料。
  5. 调用搜索或抓取工具读取 README / Install / 配置文档。
  6. 模型整理信息，决定输出结构。
  7. 如有写文件权限，通过文件工具生成 Markdown。
  8. 如果启用 sandbox，文件操作在受控环境中完成。
  9. Gateway 把过程和最终结果返回前端。
  10. 用户在页面看到报告和文件产物。

这条链路里，任何一环都可能出问题。

常见失败点

模型配置错误

表现：任务一开始就失败，或者模型列表为空。

优先检查：

  • config.yaml 中 models 是否存在；
  • .env 是否有对应 API key；
  • use、model、base_url 是否匹配；
  • make doctor 是否通过。

搜索不可用

表现：普通对话可以，研究任务失败。

优先检查：

  • 搜索 provider 是否启用；
  • .env 是否有搜索 API key；
  • 当前网络是否能访问 provider；
  • 工具结果是否真的返回了可用网页。

文件写入失败

表现：Agent 说要保存文件，但最终没有产物。

优先检查：

  • 是否开启文件写入能力；
  • sandbox 目录是否可写；
  • 文件路径是否正确；
  • 前端 artifacts 是否刷新。

Sandbox 慢或失败

表现：执行类任务长时间没有结果，或者第一次运行特别慢。

优先检查：

  • Docker 是否启动；
  • sandbox 镜像是否已拉取；
  • 当前配置使用哪种 sandbox provider；
  • 是否误启用了 provisioner 模式。

子 Agent 结果跑偏

表现：主任务方向正确，但子任务返回的材料不相关。

优先检查：

  • 子任务描述是否清楚；
  • 是否给了足够上下文；
  • 是否拆得太碎；
  • 汇总阶段是否丢失约束。

排错时怎么定位

建议按这条顺序排：

  1. 入口层：前端请求是否发出，API 是否返回。
  2. Gateway 层：后端是否收到请求，thread / run 是否创建。
  3. 模型层：模型配置和 API key 是否可用。
  4. 工具层：搜索、文件、bash、MCP 是否返回正常结果。
  5. Sandbox 层：执行环境是否启动、是否有权限。
  6. Agent 层：任务规划是否走偏，是否需要调整 prompt / skill。

不要一上来就改源码。很多问题是配置、权限、环境或 provider 差异导致的。

这一篇的完成标准

读完这一篇，你应该能把一次 DeerFlow 任务拆成几层：

  • 输入入口；
  • Gateway；
  • Agent Runtime；
  • Lead Agent；
  • 模型决策；
  • tools / skills / sub-agents；
  • sandbox 和文件产物；
  • 流式返回和前端展示。

后面如果继续写工具、skills、sub-agents、MCP，每一篇都可以放回这条链路里理解。
