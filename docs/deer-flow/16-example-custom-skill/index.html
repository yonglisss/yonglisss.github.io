16. 终局实战三：给研究型 Agent 写一个 Skill

设计一个 research-report Skill，用稳定流程约束 DeerFlow 研究型 Agent 的资料收集、证据判断、报告生成和文件验证。

为什么这一步要写 Skill

上一篇已经设计了任务流。

但如果只靠用户 prompt，Agent 仍然可能每次执行不一致：有时只看搜索摘要，有时忘记抓正文，有时不保存文件，有时不写不确定性。

Skill 的价值就是把这条研究流程固化下来。

这篇设计一个最小可用的：

research-report Skill

目标：约束研究型 Agent 的工作方法。

Skill 的目标

这个 Skill 要让 Agent 在研究任务中做到：

  • 先理解研究问题；
  • 必要时澄清；
  • 优先查一手资料；
  • 搜索后抓取正文；
  • 区分事实、判断和不确定性；
  • 生成 Markdown 报告；
  • 保存文件；
  • 写完后读取验证；
  • 不在来源不足时脑补。

这不是让 Agent 更会写漂亮话，而是让它少犯研究任务里的低级错误。

Skill 触发条件

Skill 应该在这些任务中触发：

研究某个主题
生成研究报告
分析某个开源项目
比较几个技术方案
总结某个框架架构
写一份带来源的 Markdown 报告

不应该在这些任务中触发：

闲聊
短问答
翻译一句话
改一个 CSS 样式
执行一个明确命令

触发条件越清楚，越不容易误用。

Skill 内容草案

可以先写成这样：

# Research Report Skill

Use this skill when the user asks for a research report, technical investigation, framework analysis, or source-grounded summary.

## Workflow
1. Restate the research question and scope.
2. If the request is ambiguous, ask one clarification question; otherwise proceed with reasonable defaults.
3. Prefer official and primary sources before secondary articles.
4. Use search only to discover candidate sources.
5. Fetch or read the actual source text before using it as evidence.
6. Build an evidence list before writing the report.
7. Separate facts, interpretation, and uncertainty.
8. Write the final report as Markdown.
9. Save the report to a file.
10. Read the file back and verify title, sections, sources, and missing placeholders.

## Output structure
- Title
- Executive summary
- Background
- Key findings
- Evidence and sources
- Risks and uncertainties
- Next steps

## Pitfalls
- Do not write from search snippets only.
- Do not invent sources.
- Do not hide uncertainty.
- Do not claim a file was created without verifying it.
- Do not include secrets or credentials.

这个版本已经能约束大多数研究任务。

配合工具使用

Skill 本身不执行动作，但会指导 Agent 调工具。

它会影响工具使用顺序：

search → fetch/read → evidence list → write_file → read_file → present artifact

如果接了 MCP，可以插入：

MCP knowledge search → evidence merge

如果任务复杂，可以插入：

sub-agent delegation → result synthesis

Skill 和 Tool Policy

官方 Skill 实现里有 tool_policy.py，说明 Skill 可能和工具策略有关。

研究型 Skill 推荐默认允许：

  • search；
  • fetch / crawl；
  • read_file；
  • write_file；
  • present_file；
  • 只读 MCP；
  • 必要时 sub-agent。

谨慎使用：

  • bash；
  • 写入外部系统的 MCP；
  • 发消息；
  • 修改数据库；
  • 删除文件。

第一版研究型 Agent 不需要高风险工具。

Skill 的验证标准

这个 Skill 是否有效，可以用几个问题检查：

  • Agent 是否先明确研究范围？
  • 是否优先使用官方来源？
  • 是否抓取正文，而不是只看搜索摘要？
  • 是否输出证据和来源？
  • 是否写出不确定性？
  • 是否生成文件？
  • 是否读取文件验证？
  • 是否避免编造来源？

如果这些没有做到，说明 Skill 太抽象，需要补具体步骤。

一个测试任务

可以用这个任务测试：

研究 DeerFlow 2.0 的 Agent 架构，输出一份 Markdown 报告。
要求覆盖 Gateway、Lead Agent、Tools、Sandbox、MCP、Memory。
必须基于官方 README、backend README 和当前仓库结构。

预期行为：

  1. Agent 识别这是研究报告任务；
  2. 激活 research-report Skill；
  3. 读取或抓取官方资料；
  4. 查询当前仓库结构；
  5. 生成证据列表；
  6. 写入 Markdown 文件；
  7. 读取文件检查；
  8. 返回文件路径和简短摘要。

常见问题

Skill 太长怎么办？

Skill 不是论文。先写最关键流程和坑。

Skill 要不要写很多示例？

可以写 1-2 个高质量示例，不要堆太多。

Skill 能不能替代测试？

不能。Skill 只是方法，最终仍要用工具结果、文件检查、build 或 tracing 验证。

Skill 要不要包含真实 API key？

绝对不要。文档、Skill、示例里都只能写占位符。

这一篇的完成标准

读完这一篇，你应该能做出一个最小 research-report Skill，并知道：

  • 它什么时候触发；
  • 它约束什么流程；
  • 它如何指导工具使用；
  • 它如何避免搜索摘要写报告；
  • 它如何验证最终文件产物。
