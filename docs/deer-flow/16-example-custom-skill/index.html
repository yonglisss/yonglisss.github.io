16. 实战三：给 DeerFlow 增加一个自定义 Skill

用一个真实场景演示如何定义、编写、挂载和验证 DeerFlow 自定义 Skill。

实战目标

这一篇做一个最小自定义 Skill。

场景选择“写开发者教程”。因为这个场景稳定、重复、有明确流程，也容易验证。

目标是让 DeerFlow 下次写教程时，自动遵循固定流程：先查官方资料、再写可执行步骤、最后验证构建。

先定义场景

不要一上来就写 skill 文件。

先回答几个问题：这个 skill 什么时候用？它解决什么问题？标准步骤是什么？最容易踩什么坑？完成后怎么验证？

对于开发者教程，可以这样定义：

当用户要求为开源工具写教程、配置说明、实战文档时使用。
目标是产出可执行、可验证、不过度营销的中文开发者教程。

最小 Skill 内容

一个最小 Skill 可以包含：

# Developer Tutorial Writer

## When to use
当用户要求为开源工具写教程、配置说明、实战文档时使用。

## Steps
1. 先读取官方 README、安装文档和配置示例。
2. 检查当前站点或仓库已有文档风格。
3. 正文按“问题 → 步骤 → 验证 → 常见坑”组织。
4. 不基于搜索摘要脑补技术细节。
5. 写完后运行真实检查或构建。

## Pitfalls
- 不要写成营销稿。
- 不要复制过期博客命令。
- 不要跳过验证。

具体文件格式和目录，以 DeerFlow 当前官方实现为准。先做最小可用版本，不要一次写太复杂。

放在哪里

通常可以从项目根目录的 skills/ 开始。

如果是个人或团队私有流程，更适合放在 custom/。如果是通用能力，可以考虑放到 public/。

具体路径也可能通过 DEER_FLOW_SKILLS_PATH 或配置项调整，所以写之前先看当前配置。

让 DeerFlow 识别 Skill

写好后，需要确认 DeerFlow 能发现它。

验证顺序：文件路径正确、格式符合要求、skills 加载没有报错、UI 或 API 能看到该 skill、执行相关任务时 Agent 会采用对应流程。

如果 skill 不生效，先不要改内容，先查加载路径和格式。

用一个任务验证

可以用这个任务测试：

请为 DeerFlow 的 MCP 配置写一篇入门教程，并保存为 Markdown。要求使用开发者教程写作流程。

观察它是否先查官方资料、检查已有文档风格、给出可执行命令、标注常见坑、写完后验证、避免营销表达。

调整 Skill

第一次写的 skill 通常不会完美。

调整时不要只加长文字，要补行为约束：哪一步必须做、哪一步不能做、哪些情况要停下来、验证失败怎么办、输出应该长什么样。

常见失败点

  • skill 文件路径不对；
  • 格式不符合要求；
  • 内容太空泛；
  • 没写触发条件；
  • 没写验证方式；
  • 和现有 skill 冲突；
  • Agent 没有拿到相关上下文。

复盘

自定义 Skill 的价值，是把一次成功经验变成下次可复用能力。

不要把 Skill 当成“更多 prompt”。它应该像团队内部 SOP：什么时候用、怎么做、怎么验收、哪些坑不能踩。
