14. 终局实战一：设计一个研究型 Agent

从需求和边界开始，设计一个基于 DeerFlow 的研究型 Agent：输入、输出、工具、Skill、MCP、产物和验收标准。

实战从设计开始

前面已经看完了核心能力：Tools、Skills、Sub-agents、Memory、Sandbox、MCP 和可观测性。

从这一篇开始，进入终局实战：

自己动手做一个研究型 Agent

这里不是重新造一个 DeerFlow，也不是写一个玩具 demo，而是在 DeerFlow 现有能力上设计一条稳定任务流。

目标是：用户输入一个研究主题，Agent 能收集资料、整理结构、生成 Markdown 报告，并把报告作为文件产物保存下来。

先定义这个 Agent 做什么

研究型 Agent 的最小目标：

输入：一个研究主题
输出：一份 Markdown 研究报告
过程：搜索资料 → 抓取正文 → 整理观点 → 生成报告 → 保存文件 → 返回 artifact

不要一开始就追求复杂能力。

第一版先做到：

  • 能明确研究问题；
  • 能查公开资料；
  • 能抓取关键来源；
  • 能区分事实和推断；
  • 能生成结构化 Markdown；
  • 能保存报告文件；
  • 能说明来源不足时无法下结论。

不做什么也要写清楚

一个 Agent 的边界比能力更重要。

第一版先不做：

  • 自动发布文章；
  • 修改生产系统；
  • 写数据库；
  • 自动发飞书 / Slack；
  • 使用高权限 MCP；
  • 基于搜索摘要直接写结论；
  • 在来源不足时脑补。

这些边界能显著降低失败风险。

输入格式

可以先规定一个简单输入：

请研究：DeerFlow 2.0 的 Agent 架构
输出：Markdown 报告
重点：架构、核心模块、可扩展点、风险

更好的输入模板：

研究主题：
研究目标：
必须覆盖：
不需要覆盖：
输出长度：
是否需要源码分析：
是否允许使用 MCP：

输入越清楚，后续拆任务越稳定。

输出结构

第一版 Markdown 报告建议固定结构：

# 标题

## 结论摘要

## 背景

## 关键发现

## 证据和来源

## 风险与不确定性

## 下一步建议

固定结构的好处是：

  • Skill 容易约束；
  • 产物容易检查；
  • 后续可以自动评估；
  • 用户知道会拿到什么。

工具清单

研究型 Agent 至少需要这些工具能力：

能力用途
search找候选来源
fetch / crawl读取正文
read_file读取本地资料或源码
write_file保存报告
MCP 可选查询外部知识库 / GitHub / 文档系统
present_file / artifact把报告返回给用户

注意：搜索不是证据，抓取后的正文才更接近证据。

Skill 负责约束流程

这个 Agent 需要一个 Skill 约束研究方法。

Skill 应该规定：

1. 先复述研究问题，必要时收窄范围；
2. 搜索至少多个候选来源；
3. 关键结论必须基于抓取正文；
4. 区分事实、判断和不确定性；
5. 输出 Markdown；
6. 写入文件后读取验证；
7. 来源不足时明确说明，不要脑补。

没有 Skill，Agent 很容易退化成“搜索几条摘要后直接写文章”。

Sub-agents 什么时候加入

第一版不一定要使用 Sub-agents。

只有当研究主题明显可拆时再用。

例如：

Sub-agent A：查官方资料
Sub-agent B：查源码结构
Sub-agent C：查外部文章或竞品
Lead Agent：汇总、去重、判断可信度、生成报告

如果主题很小，就不要拆。

MCP 什么时候加入

MCP 是增强项，不是第一步必须项。

适合接入：

  • 只读内部知识库；
  • GitHub 只读查询；
  • 文档系统搜索；
  • 本地索引服务。

不建议第一版接入：

  • 写数据库；
  • 修改 issue；
  • 发消息；
  • 操作生产系统。

先从只读工具开始。

产物设计

最终产物应该是文件，不只是回答。

建议输出路径类似：

outputs/research/{slug}.md

报告生成后必须做两步验证：

1. 文件是否存在
2. 读取文件确认结构和内容

如果有前端 artifact 展示能力，再把这个文件返回给用户。

验收标准

第一版研究型 Agent 的验收标准：

  • 输入一个主题能跑完；
  • 至少使用搜索和正文抓取；
  • 生成 Markdown 文件；
  • 报告包含结论、证据、风险和来源；
  • 文件能被读取；
  • 不能基于未验证来源编造结论；
  • 失败时能说明卡在哪一步。

  • 研究型 Agent 的目标；
  • 第一版不做什么；

  • 需要哪些工具；
  • Skill 如何约束流程；
  • MCP 和 Sub-agents 什么时候加入；
  • 最终产物如何验收。
