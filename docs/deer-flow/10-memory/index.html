Technical Guide

10. Memory 原理：长期上下文该记什么

理解 DeerFlow 的 Memory：它如何从对话中抽取长期上下文、如何注入系统提示，以及为什么错误记忆比没有记忆更危险。

Memory 不是聊天记录仓库

Memory 很容易被误解成“把所有历史都存下来”。

但对 Agent 系统来说，Memory 的价值不是存越多越好，而是把长期有效、会影响未来任务的上下文沉淀下来。

官方后端 README 对 Memory System 的描述是：LLM-powered persistent context retention across conversations。它会做自动抽取、结构化存储、批量更新，并把重要事实注入 Agent prompt。

也就是说，Memory 是让 Agent 在跨会话任务里更像一个持续工作的助手，而不是每次重新认识用户。

DeerFlow 里的 Memory 相关目录

官方当前仓库里，Memory 相关代码主要在：

backend/packages/harness/deerflow/agents/memory/
├── message_processing.py
├── prompt.py
├── queue.py
├── storage.py
├── summarization_hook.py
└── updater.py

还有持久化相关目录：

backend/packages/harness/deerflow/persistence/thread_meta/memory.py
backend/app/gateway/routers/memory.py

从文件名可以看出，Memory 不是单个 JSON 文件的问题，它至少包含：

  • 消息处理；
  • 记忆抽取 prompt；
  • 更新队列；
  • 存储；
  • 摘要 hook；
  • Gateway 的 memory API。

Memory 解决什么问题

长任务和多轮任务有几个典型问题：

  • 用户偏好每次都要重复说；
  • 项目固定路径每次都要重新告诉；
  • 同类任务的约定容易忘；
  • 一段会话太长，模型上下文塞不下；
  • 下一次任务无法接上长期背景。

Memory 解决的是这些长期上下文问题。

例如：

用户偏好中文、短句、少模板化。
某项目固定在 /root/code/future-observer-web。
写教程要先查官方资料，不能凭摘要脑补。
某类内容要优先走 Polymarket 数据源。

这些信息跨任务仍然有价值，适合进入 Memory。

什么值得记

适合进入 Memory 的信息通常有几个特点：

  • 长期有效；
  • 多次任务会复用；
  • 用户不希望反复说明；
  • 能明显减少沟通成本；
  • 经过确认或多次出现；
  • 不容易在几天内过期。

典型例子：

类型示例
用户偏好默认中文、回复简短、不要营销腔
项目约定某仓库路径、构建命令、提交作者要求
内容定位AI 工具号偏实操，未来观察者偏趋势分析
工作习惯大任务分批提交，不要攒到最后
稳定边界私密数据不上云，公开内容不用真实姓名

这些信息会影响未来任务，值得保存。

什么不该记

不适合进入 Memory 的信息：

  • 临时任务进度；
  • 某次 commit hash；
  • 某个 PR 编号；
  • 一次运行输出；
  • 很快会过期的链接；
  • 未验证的猜测；
  • 某次对话中的中间结论；
  • 原始大段资料。

错误记忆比没有记忆更麻烦。

比如错误记住一个项目路径，Agent 后面会反复在错误目录里工作；错误记住用户偏好，后续所有输出都会偏。

Memory 和 Skill 的区别

这两个很容易混。

Memory：长期事实
Skill：可复用流程

例子：

  • “用户偏好中文短回复”是 Memory。
  • “写开发者教程时先查官方 README，再跑 build”是 Skill。
  • “本次已经写到第 9 篇”是临时进度，不该进 Memory。

如果把流程写进 Memory，未来会变成强约束，容易干扰当前任务；如果把临时事实写进 Memory，很快就会过期。

Memory 如何参与一次任务

一次任务中，Memory 通常不是单独输出给用户看，而是作为上下文注入 Agent。

简化理解：

用户输入
+ 当前 thread 状态
+ Skill 约束
+ Memory facts
+ 上传文件
+ 工具结果
→ Lead Agent 决策

Memory 影响的是 Agent 的判断方式。

例如用户说“继续按之前风格写”，如果 Memory 里有稳定风格偏好，Agent 就能少问一次。

但 Memory 不能替代实时检查。涉及文件、命令、仓库状态时，仍然要用工具确认当前事实。

Memory 的更新不应该太频繁

官方 README 提到 Memory 有 debounced updates，会批量更新以减少 LLM 调用。

这是合理的。因为每轮都抽取和写入 Memory 会带来几个问题：

  • 成本高；
  • 噪音多；
  • 容易把临时信息写进去；
  • 写入错误后影响后续任务；
  • 多任务并发时可能冲突。

更好的策略是：只把稳定事实、明确偏好、重复出现的约定写进去。

终局实战里怎么用 Memory

最后要做的研究型 Agent，不一定一开始就需要复杂 Memory。

更稳的做法是先让它记少量长期偏好：

默认输出 Markdown
报告要区分事实、推断、观点
来源不足时停止而不是脑补
用户偏好中文、克制、实操风格

这些会影响每次研究报告的质量。

但不要把每次研究主题、搜索结果、报告正文都写进长期 Memory。那些更适合保存为 artifacts 或文件。

  • Memory 保存长期事实，不是聊天记录仓库；
