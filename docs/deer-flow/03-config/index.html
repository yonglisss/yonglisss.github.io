Technical Guide

03. 配置文件：看懂 config.yaml

拆解 DeerFlow 2.0 的核心配置：模型、API key、搜索、沙盒、运行目录、skills 和渠道入口。

为什么先看配置

DeerFlow 很多能力不是写死在代码里的，而是由 config.yaml 决定：用哪个模型、能不能联网搜索、是否启用 sandbox、skills 放在哪里、是否接入飞书或 Slack。

所以跑起来之后，第一件事不是读源码，而是看懂配置文件。

这一篇不追求覆盖每个字段，只讲最影响上手的部分。

配置文件放在哪里

推荐位置：

deer-flow/config.yaml

也就是项目根目录。

官方后端查找配置的顺序大致是：

  1. 代码里显式传入的 config_path
  2. 环境变量 DEER_FLOW_CONFIG_PATH
  3. DEER_FLOW_PROJECT_ROOT 下的 config.yaml
  4. 当前工作目录下的 config.yaml
  5. 兼容旧版本的 backend / repository root 位置

日常使用里，最简单的方式就是把它放在项目根目录。

如果你需要从其他目录启动，可以设置：

export DEER_FLOW_PROJECT_ROOT=/path/to/deer-flow

如果你想直接指定某个配置文件：

export DEER_FLOW_CONFIG_PATH=/path/to/deer-flow/config.yaml

.env 和 config.yaml 怎么分工

建议这样分：

  • config.yaml：放结构化配置，比如有哪些模型、使用哪个 provider、sandbox 模式、channels 设置。
  • .env：放密钥，比如 OPENAI_API_KEY、TAVILY_API_KEY、FEISHU_APP_SECRET。

不要把真实 key 直接写进 config.yaml 并提交。

更推荐这种写法：

models:
  - name: gpt-4o
    display_name: GPT-4o
    use: langchain_openai:ChatOpenAI
    model: gpt-4o
    api_key: $OPENAI_API_KEY

然后在 .env 里写：

OPENAI_API_KEY=your-o...-key

models：最核心的配置

DeerFlow 至少需要一个可用模型。

一个典型模型配置长这样：

models:
  - name: gpt-4o
    display_name: GPT-4o
    use: langchain_openai:ChatOpenAI
    model: gpt-4o
    api_key: $OPENAI_API_KEY

几个字段先这样理解：

字段作用
nameDeerFlow 内部识别这个模型用的名字
display_name前端或列表里展示的名字
use具体使用哪个模型适配类
modelprovider 侧真实模型名
api_keyAPI key，可以引用 .env 变量
base_urlOpenAI 兼容网关地址，可选

如果你用 OpenRouter 或其他 OpenAI 兼容服务，通常加上 base_url：

models:
  - name: openrouter-gemini-2.5-flash
    display_name: Gemini 2.5 Flash (OpenRouter)
    use: langchain_openai:ChatOpenAI
    model: google/gemini-2.5-flash-preview
    api_key: $OPENROUTER_API_KEY
    base_url: https://openrouter.ai/api/v1

重点是：use 决定调用方式，model 决定远端具体模型，api_key 和 base_url 决定连到哪里。

OpenAI Responses API

如果你想通过 OpenAI 的 /v1/responses 路线调用模型，官方 README 给出的配置思路是：继续使用 langchain_openai:ChatOpenAI，但加上 Responses API 相关字段：

models:
  - name: gpt-5-responses
    display_name: GPT-5 (Responses API)
    use: langchain_openai:ChatOpenAI
    model: gpt-5
    api_key: $OPENAI_API_KEY
    use_responses_api: true
    output_version: responses/v1

这类配置适合你明确知道 provider 支持什么接口。刚入门时，不建议同时尝试太多 provider，先让一个模型稳定可用。

本地或自托管模型

如果你用 vLLM 这类自托管服务，配置会更接近这样：

models:
  - name: qwen3-32b-vllm
    display_name: Qwen3 32B (vLLM)
    use: deerflow.models.vllm_provider:VllmChatModel
    model: Qwen/Qwen3-32B
    api_key: $VLLM_API_KEY
    base_url: http://localhost:8000/v1
    supports_thinking: true

本地模型的关键不是 DeerFlow，而是模型服务本身要先稳定：

  • vLLM 服务是否已启动
  • base_url 是否能访问
  • 模型名是否和服务端一致
  • API key 是否被服务端接受
  • reasoning / thinking 字段是否被当前模型和服务支持

如果只是学习 DeerFlow，不建议一开始就把“部署本地大模型”和“跑通 DeerFlow”绑在一起。变量太多，排错会很慢。

搜索配置：不是必需，但很常用

DeerFlow 可以接搜索和抓取能力。研究型任务、资料整理、趋势分析都依赖它。

常见做法是在 .env 里放搜索 provider 的 key：

TAVILY_API_KEY=your-t...-key
INFOQUEST_API_KEY=your-i...-key

启用哪个 provider，以 make setup 生成的配置或 config.example.yaml 为准。

这里要注意：模型能回答，不代表搜索已经可用。

你可以这样分层验证：

  1. 先问一个不需要联网的问题，验证模型链路。
  2. 再问一个必须联网的问题，验证搜索链路。
  3. 如果第 1 步成功、第 2 步失败，优先检查搜索 key 和搜索配置。

Sandbox：让执行能力有边界

DeerFlow 的重要能力之一是执行任务，而执行就会带来风险。

Sandbox 配置决定代码、命令、文件写入等能力在哪里运行。官方文档里提到的方式包括：

  • 本地执行
  • Docker / Container sandbox
  • Docker + Kubernetes provisioner

如果配置里使用类似下面的 provider：

sandbox:
  use: deerflow.community.aio_sandbox:AioSandboxProvider

并且配置了 provisioner_url，Docker 开发启动时会按对应模式拉起 provisioner。

入门阶段建议：

  • 能用 Docker sandbox 就不要直接在宿主机裸跑高风险任务。
  • 第一次使用前可以先执行 make setup-sandbox 拉镜像。
  • 不确定某个任务会不会写文件、执行命令时，先用低权限、低风险目录测试。

Sandbox 不是“绝对安全”，但它能把很多误操作挡在宿主机之外。

运行目录相关环境变量

除了模型和搜索，几个路径变量也很重要。

DEER_FLOW_PROJECT_ROOT

显式告诉 DeerFlow 项目根目录在哪里：

export DEER_FLOW_PROJECT_ROOT=/path/to/deer-flow

适合从其他目录启动，或者被脚本 / 服务管理器启动时使用。

DEER_FLOW_CONFIG_PATH

直接指定配置文件：

export DEER_FLOW_CONFIG_PATH=/path/to/config.yaml

适合多环境配置，例如开发、测试、生产分别使用不同配置。

DEER_FLOW_HOME

控制运行期状态保存位置。默认通常在项目根目录下的 .deer-flow。

如果你希望状态数据放到独立磁盘或持久化目录，可以设置：

export DEER_FLOW_HOME=/data/deer-flow-state

DEER_FLOW_SKILLS_PATH

控制 skills 的目录。默认是项目根目录下的 skills/。

export DEER_FLOW_SKILLS_PATH=/data/deer-flow-skills

当你要把 skills 当成长期资产管理时，这个变量会很有用。

Skills：把可复用能力沉淀下来

DeerFlow 里的 skills 可以理解成“可复用的工作方法和能力描述”。

它和一次性 prompt 的区别是：

  • prompt 更像临时指令；
  • skill 更像长期能力；
  • tool 更偏具体可调用动作；
  • agent 负责在任务中选择和组合这些能力。

入门阶段你不需要马上写复杂 skill，但应该知道它默认从哪里加载：

deer-flow/skills/

或者由 DEER_FLOW_SKILLS_PATH / skills.path 指定。

后续教程会专门做一篇“改一个能力”，到时再动手写。

Channels：把 DeerFlow 接进聊天入口

DeerFlow 支持多个即时通讯入口，例如：

  • Telegram
  • Slack
  • Feishu / Lark
  • 企业微信
  • 钉钉

这类配置通常会出现在 channels 下。

飞书配置大致会涉及：

channels:
  feishu:
    enabled: true
    app_id: $FEISHU_APP_ID
    app_secret: $FEISHU_APP_SECRET

对应 .env：

FEISHU_APP_ID=cli_xxxx
FEISHU_APP_SECRET=***

不建议一开始就配 IM 渠道。建议顺序是：

  1. Web 页面能跑通。
  2. 模型和搜索稳定。
  3. 文件 / sandbox 能力按预期。
  4. 再接飞书、Slack 或其他渠道。

否则排错时很难判断是 DeerFlow、模型、网络、权限还是 IM 平台的问题。

修改配置后的验证顺序

每次改完配置，建议按这个顺序检查：

make doctor

然后启动服务。

再分别验证三类任务：

  1. 不联网普通对话：确认模型可用。
  2. 联网研究任务：确认搜索可用。
  3. 文件 / 执行任务：确认 sandbox 和文件系统能力可用。

不要直接上来跑一个复杂长任务。复杂任务失败时，可能是模型、搜索、sandbox、文件权限、工具配置任意一个环节坏了。

最小可用配置心法

刚开始只追求一个目标：少配，但每一项都能验证。

建议最小闭环是：

  • 一个可用模型
  • .env 中有对应 API key
  • make doctor 通过
  • Web 页面能打开
  • 一个不联网任务能完成

然后再逐步加：

  • 搜索 provider
  • sandbox
  • skills
  • IM channels
  • tracing
  • 自托管模型

DeerFlow 的配置能力很多，但不要一次性全开。先让主链路稳定，再增加变量。
