15. 终局实战二：设计 Agent 任务流

把研究型 Agent 拆成可执行流程：澄清问题、收集来源、抓取正文、整理证据、生成报告、保存文件和返回结果。

任务流比提示词更重要

一个研究型 Agent 的质量，不取决于一句“请深入研究”。

更关键的是任务流：它每一步做什么、如何验证、什么时候停止、失败时怎么处理。

这一篇把研究型 Agent 拆成一条最小可执行流程。

总流程

建议第一版任务流如下：

1. 解析研究主题
2. 判断是否需要澄清
3. 制定研究计划
4. 搜索候选来源
5. 抓取关键正文
6. 整理证据表
7. 生成 Markdown 报告
8. 保存报告文件
9. 读取文件验证
10. 返回总结和 artifact

每一步都要有输入、输出和失败处理。

第一步：解析研究主题

用户输入通常不够结构化。

Agent 先要把它拆成：

研究对象：
研究目标：
必须覆盖：
输出形式：
限制条件：

例如：

研究对象：DeerFlow 2.0
研究目标：理解 Agent 架构
必须覆盖：Gateway、Lead Agent、Tools、Sandbox、MCP
输出形式：Markdown 报告
限制条件：必须基于官方资料和当前源码结构

这一步不要急着搜索。先把任务理解清楚。

第二步：判断是否需要澄清

如果用户输入太模糊，应该先澄清。

需要澄清的情况：

  • 主题太大；
  • 输出形式不明确；
  • 是否需要源码分析不明确；
  • 是否允许联网不明确；
  • 是否需要内部资料不明确；
  • 时间范围不明确。

不需要澄清的情况：

  • 主题明确；
  • 输出形式明确；
  • 可以用默认约定继续；
  • 澄清会明显拖慢任务。

对于自动化 Agent，建议允许默认策略：信息足够时直接推进，不要动不动打断用户。

第三步：制定研究计划

研究计划不要太长。

一个可执行计划可以是：

1. 查官方 README 和 backend README
2. 查 config.example.yaml
3. 查关键源码目录
4. 搜索外部背景资料
5. 汇总架构和风险
6. 生成 Markdown 报告

计划的作用不是展示“我很聪明”，而是给后面的工具调用建立顺序。

第四步：搜索候选来源

搜索阶段只负责找来源，不负责下结论。

候选来源优先级：

官方仓库 / 官方文档
官方 README / config 示例
源码目录
可信技术文章
社区讨论
普通博客

如果官方资料能回答问题，不要优先引用二手文章。

搜索结果要记录：

  • 标题；
  • URL；
  • 来源类型；
  • 可能相关点；
  • 是否需要抓全文。

第五步：抓取关键正文

只看搜索摘要不够。

关键来源必须进入抓取正文阶段。

抓取后要判断：

  • 内容是否完整；
  • 是否和问题相关；
  • 是否是一手来源；
  • 是否过期；
  • 是否和其他来源冲突。

如果正文拿不到，就不能把它当作强证据。

第六步：整理证据表

在写报告前，先整理证据。

可以让 Agent 内部形成这样的表：

结论：DeerFlow 使用 Gateway 统一后端入口
证据：backend README 架构图提到 Gateway API，GitHub tree 中存在 backend/app/gateway
来源：backend/README.md、GitHub tree
可信度：高

这样写报告时就不会只靠感觉。

第七步：生成 Markdown 报告

报告结构固定：

# 研究标题

## 结论摘要

## 背景

## 关键发现

## 证据和来源

## 风险与不确定性

## 下一步建议

固定结构能降低输出漂移。

其中“风险与不确定性”很重要。Agent 不应该把所有内容都写成确定事实。

第八步：保存报告文件

报告应该写入文件，而不是只输出在聊天框。

推荐文件名规则：

outputs/research/{date}-{slug}.md

写入前要确认目录。写入后要读取验证。

如果文件系统或 sandbox 不可用，就应该明确报告不能落盘，而不是假装已生成。

第九步：读取文件验证

写完文件后，至少检查：

  • 文件存在；
  • 标题正确；
  • 章节齐全；
  • 来源列表存在；
  • 没有明显占位符；
  • 没有泄露密钥；
  • Markdown 格式正常。

这一步很重要。

很多 Agent 会说“文件已生成”，但实际路径错了、文件为空、格式坏了。验证可以避免这种假成功。

第十步：返回结果

最后返回给用户的内容应该短：

已完成报告：xxx.md
覆盖：A/B/C
主要结论：...
不确定性：...

如果有 artifacts，就附上文件。

不要把整篇报告再复制一遍，除非用户要求。文件产物才是最终交付。

失败处理

任务流必须允许失败。

常见失败处理：

失败点处理方式
搜索为空换 query，仍为空就说明无法完成
抓取失败换来源或标记来源不可用
来源冲突在报告里写明冲突，不强行统一
文件写入失败返回错误和路径，不声称成功
MCP 不可用降级为公开资料，但标注缺口
token 过长外部化工具输出，分阶段总结

好的 Agent 不是永不失败，而是失败时知道停在哪里。

这一篇的完成标准

读完这一篇，你应该能把研究型 Agent 拆成一条可执行链路：

解析 → 澄清 → 计划 → 搜索 → 抓取 → 证据 → 报告 → 保存 → 验证 → 返回
