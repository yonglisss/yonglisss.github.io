17. 终局实战四：接入一个只读 MCP 数据源

给研究型 Agent 增加一个低风险 MCP：先验证 server、配置、工具加载、cache、权限，再把结果并入研究报告。

MCP 不要一上来接高风险系统

研究型 Agent 可以只用公开搜索完成第一版。

但如果你希望它查询内部知识库、GitHub 仓库、文档系统或本地索引，就需要 MCP。

这一篇只做低风险设计：接入一个只读 MCP 数据源。

不要一开始就接能写数据库、发消息、改 issue、操作生产系统的 MCP。先从只读开始。

推荐接入目标

适合第一版的 MCP：

  • 只读知识库搜索；
  • GitHub 仓库只读查询；
  • 文档系统只读搜索；
  • 本地资料索引；
  • 只读数据库查询。

不建议第一版接：

  • 写数据库；
  • 发飞书 / Slack；
  • 修改 GitHub issue；
  • 删除文件；
  • 触发部署；
  • 操作云资源。

研究型 Agent 的 MCP 目标是补充资料，不是自动执行高风险动作。

DeerFlow 的 MCP 接入点

前面已经看过，官方当前目录里 MCP 相关实现主要在：

deerflow/mcp/client.py
deerflow/mcp/session_pool.py
deerflow/mcp/oauth.py
deerflow/mcp/cache.py
deerflow/mcp/tools.py

Gateway 层也有：

backend/app/gateway/routers/mcp.py

后端 README 提到的接口包括：

GET/PUT /api/mcp/config
POST /api/mcp/cache/reset

这说明 MCP 配置、工具缓存和工具加载都可以通过 Gateway 管理。

接入前先做 5 个检查

在 DeerFlow 里配置 MCP 前，先确认：

1. MCP server 能独立启动
2. server 暴露的工具是只读的
3. 需要的 token / OAuth 已准备好
4. 工具 schema 清楚，输入输出稳定
5. 工具返回内容不会过大

如果 server 自己都跑不稳，不要先怪 DeerFlow。

配置后先看工具是否加载

配置完成后，不要直接跑复杂研究任务。

先验证：

1. Gateway 能读取 MCP config
2. MCP client 能连接 server
3. tools 能被列出来
4. cache 没有旧数据干扰
5. Agent 能看到工具名

如果修改了 MCP server 或工具 schema，记得 reset cache。

最小调用测试

先让 Agent 做一个非常小的调用：

请调用知识库 MCP，搜索 “DeerFlow Gateway”，只返回前 3 条标题和来源，不要写报告。

这个测试能验证：

  • 工具是否可见；
  • 入参是否正确；
  • 权限是否够；
  • 返回结构是否可读；
  • Agent 是否会过度发挥。

如果最小调用不稳定，不要继续做完整报告。

把 MCP 结果并入研究流程

MCP 接入成功后，研究型 Agent 的流程可以变成：

解析研究主题
  ↓
公开搜索官方资料
  ↓
抓取正文
  ↓
调用只读 MCP 查询内部资料 / 仓库 / 文档
  ↓
合并证据
  ↓
生成 Markdown 报告
  ↓
保存并验证文件

注意：MCP 结果也需要验证。

它不能因为来自内部系统就自动可信。要记录来源、时间、工具名和关键字段。

报告中如何标注 MCP 来源

建议在报告里把来源分开：

## 来源

### 官方资料
- ...

### 外部公开资料
- ...

### MCP 查询结果
- 工具：knowledge_search
- 查询：DeerFlow Gateway
- 返回时间：...
- 关键条目：...

这样用户能判断哪些结论来自公开资料，哪些来自内部资料。

常见失败点

MCP 接入常见问题：

  • server 没启动；
  • 配置路径错误；
  • OAuth / token 无效；
  • 工具 schema 太复杂；
  • 工具返回过大；
  • cache 没刷新；
  • Agent 看得到工具但不会正确调用；
  • 工具返回字段模型理解错；
  • MCP 结果和公开资料冲突。

排错顺序：

server → config → auth → tool list → cache → minimal call → Agent integration → report output

和 Skill 配合

上一篇的 research-report Skill 应该加一条规则：

如果使用 MCP 查询内部资料，必须在报告中单独标注 MCP 来源；不要把 MCP 返回内容和公开来源混为一谈。

如果 MCP 不可用，Agent 应该降级：

MCP 查询失败，本报告仅基于公开资料和官方仓库。

不要因为 MCP 挂了就编造内部资料。

这一篇的完成标准

读完这一篇，你应该知道：

  • 第一版 MCP 应该选只读、低风险工具；
  • 接入前要先验证 server；
  • 配置后要验证 tools 是否加载；
  • cache 可能影响工具可见性；
  • MCP 结果要单独标注来源；
  • MCP 不可用时要降级，而不是脑补。
