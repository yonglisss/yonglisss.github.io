05. 项目结构：从目录看懂 DeerFlow

通过 DeerFlow 2.0 的仓库目录理解前端、Gateway、Agent Runtime、Sandbox、Skills、MCP 和配置系统的分工。

为什么要先看目录

跑通 DeerFlow 之后，很多人会直接去读 Agent 代码。但更稳的方式是先看目录。

一个工程化 Agent 项目，目录本身就能告诉你很多信息：

  • 用户入口在哪里；
  • 后端服务怎么组织；
  • Agent runtime 在哪里；
  • tools、skills、memory、sandbox 分别归谁管；
  • 配置文件如何影响运行行为；
  • 前端和后端之间怎么连接。

先把地图看清楚，再读源码会轻很多。

整体结构

根据官方仓库当前结构，可以先把 DeerFlow 看成这几块：

deer-flow/
├── Makefile
├── config.yaml
├── config.example.yaml
├── extensions_config.json
├── backend/
├── frontend/
└── skills/

每一层的作用大致是：

目录 / 文件作用
Makefile根目录命令入口，比如检查、安装、启动、Docker 相关命令
config.yaml本地实际运行配置，通常不提交
config.example.yaml官方配置模板，字段最完整
extensions_config.json扩展配置，常见于 MCP servers 和 skills 相关配置
backend/后端、Gateway、Agent runtime、tools、sandbox、memory 等核心逻辑
frontend/Web 前端界面
skills/Agent 可加载的 skills，包含公开 skill 和自定义 skill

不要一开始就把所有目录都打开。先抓住主线：用户从前端或 API 发起任务，后端 Gateway 接住请求，Agent runtime 执行任务，必要时调用工具、沙盒、skills 和子 Agent。

根目录：配置和命令入口

根目录最重要的是两个东西：命令和配置。

Makefile

日常命令基本从这里进：

make check
make install
make dev
make docker-init
make docker-start
make up
make down
make doctor

你可以把根目录 Makefile 理解成 DeerFlow 的操作面板。不要一上来自己拼命令，优先看官方 Makefile 暴露了什么入口。

config.yaml

这是本地实际生效的配置文件。

它决定：

  • 用哪些模型；
  • 搜索 provider 怎么配；
  • sandbox 用什么模式；
  • skills 从哪里加载；
  • channels 是否启用；
  • tracing 是否打开；
  • token budget 等运行限制是否开启。

如果 DeerFlow 行为和你预期不一致，第一步不是改代码，而是先查 config.yaml。

config.example.yaml

这是配置参考模板。

当你不知道某个字段怎么写时，不要凭感觉猜，先看这里。它通常比 README 更完整，尤其是模型 provider、sandbox、channels、token budget 这类细节。

frontend/：Web 界面

frontend/ 是 DeerFlow 的前端应用。

你通过浏览器访问的页面，通常来自这里。它负责：

  • 展示对话和任务状态；
  • 发送用户输入；
  • 展示工具调用、文件、产物；
  • 和后端 Gateway 通信。

学习 DeerFlow 的第一阶段，不需要深读前端。你只要知道：前端不是 Agent 本体，它是用户入口和状态展示层。

如果你以后要改 UI、加按钮、展示更多运行信息，再进入 frontend/。

backend/：核心在这里

backend/ 是 DeerFlow 最值得看的部分。

官方开发说明里把它分成两层：

backend/
├── app/
└── packages/

可以先这样理解：

  • app/ 更偏应用层，比如 Gateway API、channels；
  • packages/harness/ 更偏 Agent 运行框架本体。

backend/app/gateway/：统一后端入口

Gateway 是 DeerFlow 的后端 API 入口。

它负责接收前端、API 或部分渠道层的请求，再把任务交给 Agent runtime。

官方说明里提到，Gateway API 默认端口是 8001。在完整运行时，Nginx 会把统一入口转发到 Gateway，对外常见访问地址是：

http://localhost:2026

你可以把 Gateway 理解成前端和 Agent runtime 之间的服务层。

它不是只做转发，通常还会处理：

  • models 接口；
  • threads / runs；
  • uploads；
  • artifacts；
  • memory；
  • skills；
  • MCP；
  • channels 相关接口。

如果你想知道“前端点一次发送后，请求进了哪里”，先看 Gateway。

backend/app/channels/：IM 入口

DeerFlow 支持 Telegram、Slack、飞书、企业微信、钉钉等 IM 渠道。

这类逻辑通常在 channels 相关目录和配置里。

它的作用是把不同平台的消息转成 DeerFlow 可以处理的任务，再把结果发回对应会话。

入门阶段不建议先接 IM。因为它会额外引入平台权限、token、事件订阅、网络连接等变量。先让 Web 入口稳定，再接 channels 更好排错。

backend/packages/harness/：Agent 系统本体

这是理解 DeerFlow 的关键目录。

官方说明里提到，Python 包导入路径主要是：

deerflow.*

目录大致长这样：

backend/packages/harness/deerflow/
├── agents/
├── sandbox/
├── subagents/
├── tools/
├── mcp/
├── models/
├── skills/
├── config/
├── community/
├── reflection/
├── utils/
└── client.py

这基本就是 DeerFlow 的能力地图。

agents/：主 Agent 和中间件

agents/ 里是 Agent 系统的核心。

里面通常会包含：

  • lead_agent/：主 Agent；
  • middlewares/：中间件；
  • memory/：记忆提取和注入相关逻辑；
  • thread_state.py：线程 / 会话状态结构。

如果你想理解一次任务为什么这样执行，最终还是要回到这里。

但第一次读源码时，不建议直接从最深处开始。先读目录，再读执行链路，再读关键模块，会更顺。

models/：模型适配层

models/ 负责把不同 provider 接进 DeerFlow。

这也是为什么 config.yaml 里的 use 字段很重要。它通常指向某个模型适配类，例如：

use: langchain_openai:ChatOpenAI

或者：

use: deerflow.models.vllm_provider:VllmChatModel

不同模型的差异不只是 API key 和 model name。还可能包括：

  • 是否支持 vision；
  • 是否支持 thinking / reasoning；
  • 是否支持 Responses API；
  • tool call 字段是否有 provider 特殊格式；
  • 多轮工具调用时是否需要保留额外字段。

所以模型配置异常时，不要只盯着 key，也要看 provider 适配是否匹配。

tools/ 和 community/：可调用能力

tools/ 放内置工具，community/ 放一些社区工具或 provider 相关能力。

常见能力包括：

  • 文件展示；
  • 图片查看；
  • 用户澄清；
  • Tavily 搜索；
  • Jina 抓取；
  • Firecrawl；
  • image search；
  • AIO sandbox provider。

工具目录的重点是：模型不会自己“真的做事”，它必须通过这些工具和外部世界交互。

如果你想给 DeerFlow 增加一个新能力，后面大概率会在 tools、skills 或 MCP 之间做选择。

skills/：技能加载和解析

backend/packages/harness/deerflow/skills/ 是 skill 的系统逻辑。

而根目录下的：

skills/

通常是具体 skill 文件所在的位置。

这两个不要混淆：

  • deerflow/skills/：负责发现、加载、解析 skill；
  • 根目录 skills/：放实际可被 Agent 使用的 skill 内容。

如果你只是写一个自定义 skill，通常先动根目录的 skills/custom/。如果你要改 skill 机制，才进入后端包里的 skills/。

sandbox/：执行环境边界

sandbox/ 负责执行环境。

官方结构里能看到：

sandbox/
├── local/
├── sandbox.py
├── tools.py
└── middleware.py

可以先这样理解：

  • sandbox.py：抽象接口；
  • local/：本地文件系统 provider；
  • tools.py：bash、ls、read/write、str_replace 等执行和文件工具；
  • middleware.py：sandbox 生命周期管理。

这部分和安全强相关。任何涉及命令执行、文件写入、代码运行的能力，都应该和 sandbox 一起看。

subagents/：子 Agent 系统

subagents/ 负责子 Agent 的注册和执行。

官方结构里包括：

subagents/
├── builtins/
├── executor.py
└── registry.py

含义大致是：

  • builtins/：内置子 Agent；
  • registry.py：子 Agent 注册表；
  • executor.py：后台执行引擎。

如果你想理解“主 Agent 怎么把任务交给子 Agent”，这里是重点目录。

mcp/：外部工具协议

MCP 是接入外部工具和服务的重要方式。

mcp/ 目录通常会涉及：

  • MCP client；
  • MCP tools；
  • 工具缓存；
  • OAuth 或连接配置。

如果你已经有一个 MCP server，想让 DeerFlow 调用它，就需要看 MCP 配置和这个目录。

但入门阶段还是建议先理解内置 tools，再接 MCP。MCP 会额外带来服务启动、网络连接、权限和协议调试成本。

memory/：记忆系统

Memory 相关逻辑一部分在 agents/memory/ 下。

它负责：

  • 从对话或任务中提取可记忆信息；
  • 管理记忆写入队列；
  • 在后续任务中注入相关记忆；
  • 控制哪些信息该保留，哪些不该保留。

Agent 有记忆后，能力会变强，但也更容易被错误信息污染。所以 memory 不是“越多越好”，而是要控制质量。

client.py：嵌入式 Python Client

DeerFlow 还提供了 embedded Python client。

这意味着你不一定只能通过 Web 页面使用它，也可以在 Python 代码里把它当成能力调用。

如果你想把 DeerFlow 接进自己的自动化系统、脚本或后端服务，可以关注 client.py。

推荐读源码顺序

不要按目录字母顺序读。

更推荐：

  1. 根目录 README.md / README_zh.md：确认官方定位和启动方式。
  2. config.example.yaml：看能力开关和配置结构。
  3. backend/CLAUDE.md：看官方写给代码助手的架构说明。
  4. backend/app/gateway/：看请求入口。
  5. backend/packages/harness/deerflow/agents/：看 Agent 主流程。
  6. sandbox/、tools/、skills/、subagents/：按你想改的能力逐个深入。

这样读会比直接冲进 Agent 代码更稳。

这一篇的完成标准

读完这一篇，你应该能回答几个问题：

  • Web 页面和 Agent 本体分别在哪；
  • Gateway 在链路里做什么；
  • config.yaml 为什么比很多源码更先需要看；
  • tools、skills、sandbox、subagents、mcp 分别在哪类目录；
  • 想改 UI、模型、工具、skill、IM 渠道时，大概要从哪里入手。
