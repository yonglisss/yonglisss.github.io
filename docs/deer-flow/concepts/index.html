04. 核心概念：DeerFlow 到底在调度什么

理解 DeerFlow 2.0 的核心概念：super agent harness、tools、skills、sub-agents、memory、sandbox 和 context engineering。

先不要被名词带偏

DeerFlow 里有很多听起来很大的词：Super Agent、Sub-agents、Memory、Sandbox、Skills、Context Engineering。

如果一开始就逐个查定义，很容易变成概念堆叠。更实用的理解方式是：DeerFlow 不是在证明“多 Agent 很先进”，它真正做的是把一次复杂任务需要的几个工程能力放在同一个系统里。

一次长任务通常需要这些东西：

  • 一个主控，负责理解目标和拆步骤；
  • 一批工具，负责搜索、读写文件、执行命令；
  • 一个隔离环境，避免执行代码时污染宿主机；
  • 一套上下文管理方式，避免模型在长任务中丢失重点；
  • 一些可复用经验，让同类任务下次不用重新写提示词；
  • 一个入口，让用户可以从 Web 或 IM 工具发起任务。

DeerFlow 的核心概念，基本都围绕这几件事展开。

Super Agent Harness 是什么

官方把 DeerFlow 定位成 super agent harness。

这里的重点不是“super”，而是 “harness”。它更像一个 Agent 运行框架：把模型、工具、沙盒、记忆、子 Agent、前端和消息入口装配起来，让它们能稳定协作。

你可以这样理解：

  • 普通聊天机器人：用户问一句，模型答一句。
  • 简单 Agent：模型可以调用几个工具。
  • DeerFlow 这类 harness：不仅能调工具，还要管理任务状态、执行环境、文件产物、子任务、长期记忆和外部入口。

所以学习 DeerFlow 时，不要只问“它用了几个 Agent”。更应该问：

  • 它怎么让任务持续执行？
  • 它怎么控制工具和文件权限？
  • 它怎么把上下文传给下一步？
  • 它怎么让同一套能力可以从 Web、API 或聊天工具入口调用？

这些才是工程价值。

Lead Agent：任务的主控

Lead Agent 可以理解成主控 Agent。

它通常负责：

  • 理解用户输入；
  • 判断任务是否需要工具；
  • 拆分步骤；
  • 决定是否调用子 Agent；
  • 汇总中间结果；
  • 最后把结果返回给用户。

如果把 DeerFlow 看成一个团队，Lead Agent 就是项目负责人。它不一定亲自做所有事，但要决定下一步该交给谁、调用什么能力、什么时候收束。

这也是为什么模型质量很重要。主控模型如果判断能力弱，后面的工具和子 Agent 再多，也容易走偏。

Tools：让模型能做事

Tools 是 Agent 对外部世界的操作接口。

没有 tools，模型只能生成文本。有了 tools，模型才能做这些事：

  • 搜索网页；
  • 抓取网页内容；
  • 查看文件；
  • 写入文件；
  • 执行 bash 命令；
  • 调用 MCP server；
  • 读取图片或处理上传文件。

学习 tools 时，重点不是背有哪些工具，而是理解一个原则：

工具越强，风险越高；工具越弱，任务越难真正完成。

比如搜索工具让研究任务变强，但也会带来来源质量问题。bash 和文件写入让 Agent 能完成真实任务，但也会带来误删、误改、越权执行的风险。

所以 DeerFlow 才需要配置和 sandbox 来控制边界。

Skills：把经验沉淀成能力

Skill 可以先理解成“可复用的工作方法”。

它和 prompt 很像，但不是一次性的聊天指令。一个好的 skill 通常会告诉 Agent：

  • 什么时候使用这项能力；
  • 这类任务应该按什么步骤做；
  • 有哪些常见坑；
  • 如何验证结果；
  • 必要时可以附带脚本、模板、参考资料。

Skill 和 Tool 的区别可以这样看：

类型更像什么解决什么问题
Prompt临时指令这一次怎么做
Skill可复用方法同类任务以后怎么做
Tool可调用动作具体执行搜索、写文件、跑命令

举个例子：

  • “帮我写一篇 DeerFlow 教程”是 prompt。
  • “写开发者教程时先查官方文档、再写可验证步骤、最后跑 build”是 skill。
  • “读取文件、搜索网页、执行 npm run build”是 tool。

DeerFlow 把 skills 放进系统，是为了让 Agent 不只靠临时提示词工作，而是能复用稳定流程。

Sub-agents：把复杂任务拆出去

Sub-agents 是子 Agent。

它们适合处理相对独立的子任务，比如：

  • 一个 Agent 负责搜索资料；
  • 一个 Agent 负责读源码；
  • 一个 Agent 负责写报告；
  • 一个 Agent 负责检查结果。

但多 Agent 不是越多越好。

拆子 Agent 的前提是任务真的可以并行或分工。如果任务本身很短，硬拆反而会增加沟通成本，让结果更不可控。

一个简单判断标准是：

  • 任务需要不同视角，适合拆；
  • 任务有多个独立资料源，适合拆；
  • 任务很长，需要分阶段处理，适合拆；
  • 任务只是改一行配置，不适合拆。

学习 DeerFlow 的 Sub-agents，重点是看它怎么注册、怎么调用、怎么汇总结果，而不是只看“有几个角色”。

Memory：让任务不只活在当前上下文里

Memory 解决的是长期上下文问题。

模型的一轮上下文再长，也不适合塞进所有历史。Memory 的价值是把一些稳定信息保留下来，让后续任务能复用。

适合进入 memory 的信息通常是：

  • 用户偏好；
  • 稳定的项目约定；
  • 常用路径；
  • 重复出现的工作流程；
  • 长期有效的环境事实。

不适合进入 memory 的信息是：

  • 临时任务进度；
  • 某次运行的中间结果；
  • 很快会过期的链接或数据；
  • 尚未验证的猜测。

Memory 很有用，但也容易污染系统。错误记忆比没有记忆更麻烦，所以要谨慎。

Sandbox：给执行能力加边界

一旦 Agent 能执行命令、写文件、跑代码，就必须考虑安全边界。

Sandbox 的作用，是把执行环境和宿主机隔开。DeerFlow 支持本地执行、Docker / Container sandbox，以及更复杂的 provisioner 模式。

入门阶段可以先记住这几点：

  • 只聊天，不一定需要 sandbox；
  • 要执行代码、写文件，最好启用 sandbox；
  • sandbox 不是绝对安全，但比直接在宿主机裸跑更可控；
  • Docker 模式第一次可能慢，主要是镜像拉取和服务启动成本。

如果你要把 DeerFlow 用在长期运行或团队环境，sandbox 不是可选项，而是必须认真设计的边界。

Context Engineering：控制模型看到什么

Context Engineering 可以理解成“上下文工程”。

很多 Agent 失败，不是因为模型完全不会，而是因为它在某一步看到的信息不对：

  • 重要约束没带上；
  • 无关内容太多；
  • 工具结果没有整理；
  • 历史消息太长，重点被淹没；
  • 子任务结果没有正确汇总。

DeerFlow 这类系统需要处理的不只是 prompt，而是整个任务过程中的信息流。

所以看它的架构时，要关注：

  • 用户输入怎么变成任务上下文；
  • 工具返回结果怎么进入下一轮；
  • 子 Agent 的结果怎么汇总；
  • memory 什么时候被注入；
  • 文件和产物怎么被引用。

这部分决定了长任务能不能稳定执行。

File System：产物要落地

很多 AI 应用只停留在聊天窗口里。DeerFlow 更强调任务产物。

文件系统能力让 Agent 可以：

  • 读取项目文件；
  • 生成报告；
  • 修改代码；
  • 保存中间产物；
  • 把输出变成可复用文件。

这也是为什么 DeerFlow 更接近“工作台”而不是“聊天框”。

但文件能力同样需要约束。写文件之前要知道写到哪里、会不会覆盖已有内容、是否需要用户确认。

这些概念之间的关系

可以把 DeerFlow 的核心关系简化成这样：

用户任务
  ↓
Lead Agent 理解目标、规划步骤
  ↓
按需调用 Tools / Skills / Sub-agents
  ↓
在 Sandbox 和 File System 中执行、读写、生成产物
  ↓
Memory 和 Context Engineering 维持长期状态和上下文质量
  ↓
通过 Web / API / IM Channels 返回结果

这个图不完整，但足够帮助你建立第一层理解。

后面继续读项目结构和执行链路时，就可以把每个目录、每个配置项放回这张图里，而不是孤立地看源码。

这一篇的完成标准

读完这一篇，你不需要记住所有实现细节。

只要能分清下面几组关系，就可以继续往下：

  • Tool 负责执行动作，Skill 负责复用方法。
  • Lead Agent 负责主控，Sub-agents 负责分工。
  • Memory 保存长期信息，Context Engineering 管理当前任务信息流。
  • Sandbox 控制执行边界，File System 承载任务产物。
