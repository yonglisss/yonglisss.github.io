Technical Guide

19. 进阶路线：从使用者到改造者

收束 DeerFlow 教程：如何从能跑、会配置、看懂架构，继续走到改 Tools、写 Skills、接 MCP、做自己的 Agent。

不要一开始就冲源码

DeerFlow 这类项目很容易让人产生源码焦虑。

目录多、能力多、概念多：Gateway、runtime、Lead Agent、middleware、tools、skills、sub-agents、memory、sandbox、MCP、tracing、artifacts。

如果一上来就试图读完所有源码，很快会迷路。

更稳的路线是：

跑起来 → 会配置 → 看懂链路 → 按能力模块拆 → 做一个自己的 Agent

这套教程就是按这个顺序设计的。

第一阶段：使用者

目标：能跑、能配置、能完成基本任务。

你应该掌握：

  • 启动 DeerFlow；
  • 配置模型 provider；
  • 配置搜索能力；
  • 理解 config.yaml；
  • 知道 sandbox 模式；
  • 跑通一个简单任务；
  • 看懂基本错误。

对应教程：

01 教程总览
02 快速开始
03 配置文件

完成标准：你能让 DeerFlow 在本地或 Docker 环境里稳定启动，并完成一次简单研究任务。

第二阶段：架构理解者

目标：知道一次任务怎么流动。

你应该掌握：

  • Gateway 是入口；
  • thread / run 是任务容器；
  • runtime 组织执行；
  • middleware 准备上下文；
  • Lead Agent 决策下一步；
  • tools / sandbox / MCP 执行动作；
  • artifacts 承载产物；
  • streaming 返回过程。

对应教程：

05 项目结构
06 执行链路

完成标准：你能解释“用户输入一句话后，DeerFlow 大概经过哪些层”。

目标：能改一个能力，而不是只会使用。

先从低风险能力开始：

原理 → 源码 → 小实战

本轮先完成原理和终局实战主线。后续如果继续扩展，可以补：

目标：做一个自己的研究型 Agent。

完成标准：

  • 输入一个研究主题；
  • Agent 能收集资料；
  • 关键来源能抓正文；
  • Skill 能约束输出质量；
  • 可选 MCP 能补充资料；
  • 报告能保存为 Markdown 文件；
  • 结果能验证；
  • 失败时能定位在哪一层。

这比“看懂 DeerFlow”更重要。最终目标是能用 DeerFlow 做东西。

最重要的学习顺序

如果你只记住一条路线，就记这个：

先跑通
再看配置
再看执行链路
再按能力模块拆
最后做一个自己的 Agent

不要反过来。

很多人学 Agent 框架的问题是：源码没读懂，系统没跑起来，最后只剩概念。

这套教程希望反过来：先拿到一个能运行的系统，再逐层打开。

如何判断自己已经掌握

你可以用这组问题自测：

  • 我能从零启动 DeerFlow 吗？
  • 我知道模型、搜索、sandbox 配置在哪里吗？
  • 我能解释一次 run 的执行链路吗？
  • 我知道 Tool、Skill、MCP、Sub-agent 的区别吗？
  • 我知道 Memory 该记什么、不该记什么吗？
  • 我知道文件产物为什么要通过 artifacts 管理吗？
  • 我能为一个研究任务设计 Skill 吗？
  • 我能接一个只读 MCP 吗？
  • 我能生成并验证 Markdown 报告吗？
  • 我能在任务失败时定位是哪一层出问题吗？

如果这些能回答，说明你已经不只是“会用”，而是开始具备改造能力。

最后的建议

DeerFlow 这类项目的价值，不在于背完所有概念。

真正有价值的是：

把 Agent 当成工程系统来理解
把能力拆成模块
把每个模块接到可验证动作
最后做出一个稳定产物

所以这套教程的最终落点不是“读完 19 篇”，而是你能自己做出一个研究型 Agent，并知道下一步该改哪里。
