Technical Guide

18. 部署：从本地跑到可长期使用

从资源、Docker、配置、密钥、日志、更新和备份角度梳理 DeerFlow 长期运行部署注意事项。

本地跑通不等于可以长期运行

make dev 能跑起来，只说明开发环境可用。

如果要让 DeerFlow 长期服务自己或团队，还要考虑资源、配置、密钥、安全、日志、升级和备份。

这一篇不是完整运维手册，只给部署前必须想清楚的主线。

先选部署形态

常见选择：

形态适合场景
本地开发改源码、调试、个人体验
Docker 开发本地稳定体验、接近真实环境
生产 Docker长期运行、固定入口、统一配置
Kubernetes / Provisioner团队共享、多任务、隔离要求高

个人学习阶段不需要一上来 Kubernetes。先 Docker 跑稳更重要。

资源规划

官方 README 给了资源建议。可以按这个思路理解：

  • 本地体验：至少要能跑前端、后端、nginx 和外部模型调用；
  • Docker 开发：还要考虑镜像、容器、sandbox；
  • 长期服务：要考虑多任务、报告生成、文件产物和日志。

如果还要本机部署大模型，资源要单独算，不要把模型服务和 DeerFlow 混在一个预算里。

配置管理

长期运行时，config.yaml 要当成重要配置资产管理。

建议：不提交真实密钥，用 .env 或部署平台 secret 管理 key，开发、测试、生产分开，记录配置变更，升级前备份配置，关注 config_version。

如果官方 config.example.yaml 更新了，应该用 make config-upgrade 或官方推荐方式合并新字段，而不是手工乱补。

密钥管理

不要把 API key 写进仓库。

常见密钥包括模型 provider key、搜索 provider key、MCP server token、IM 渠道 secret、tracing 平台 key。

团队环境要控制谁能读取这些 key。Agent 执行环境也不应该默认拿到所有密钥。

网络和入口

如果只是本机访问，默认本地端口就够。

如果要对外开放，需要考虑 HTTPS、登录认证、访问控制、反向代理、防止公网直接暴露高风险接口、IM channel 的平台权限。

不要把带 bash、文件写入、MCP 写入工具的 Agent 无认证暴露到公网。

日志和可观测性

长期运行必须能查问题。

至少要能看到 Gateway 日志、Agent runtime 错误、工具调用失败、sandbox 启动失败、模型 provider 报错、token 和成本趋势。

复杂任务建议配置 LangSmith / Langfuse 之类 tracing。

文件和状态备份

DeerFlow 运行期状态默认可能在项目根目录下 .deer-flow，也可以通过 DEER_FLOW_HOME 改位置。

长期运行时要考虑状态数据放哪里、生成文件放哪里、skills 放哪里、是否需要备份、升级会不会覆盖、多用户是否隔离。

配置和 skills 都是资产，不要只备份源码。

更新版本

升级前建议：备份 config.yaml、.env、skills、运行期数据；阅读官方更新说明；检查 config.example.yaml 的 config_version；在测试环境先跑；运行 make doctor；验证模型、搜索、sandbox、文件写入、MCP。

不要在生产环境直接拉最新代码重启。

常见部署问题

  • Docker daemon 权限不足；
  • 端口被占用；
  • .env 没被加载；
  • 配置文件路径不对；
  • sandbox 镜像没拉；
  • 模型 API key 过期；
  • 搜索 provider 不可用；
  • 反向代理路径配置错误。

排查时按层看：服务是否启动、配置是否加载、模型是否可用、工具是否可用、前端是否能连 Gateway。

完成标准

一个可长期使用的部署，至少应该满足：服务可重启、配置可恢复、密钥不入库、日志可查询、文件和状态有备份、高风险工具有权限边界、升级有回滚方案。
