17. 实战四：接入一个 MCP Server

用最小路径演示 MCP Server 接入思路：选择服务、配置连接、验证工具、执行任务和排错。

实战目标

这一篇做一个 MCP 接入演练。

不限定具体 MCP server，因为不同环境可用服务不同。重点是掌握验证顺序：先让 server 单独可用，再让 DeerFlow 发现工具，最后让 Agent 调用。

选择一个简单 MCP Server

第一次不要选太复杂的服务。

更适合：只读工具、配置少、不需要复杂 OAuth、工具 schema 清楚、有明确测试命令。

不建议第一次就接生产数据库、部署系统、消息发送工具。这些副作用太强。

准备配置

通常需要确认：server 启动方式、传输方式是 stdio、HTTP 还是 SSE、是否需要 token、环境变量放哪里、DeerFlow 的 MCP 配置文件或 config.yaml 中对应字段怎么写。

具体字段以官方文档和当前仓库配置为准。

第一步：单独启动 Server

不要一开始就放进 DeerFlow。

先确认 MCP server 自己能启动，日志没有报错，依赖安装完整，token 有效。

如果 server 自己都起不来，DeerFlow 里一定也用不了。

第二步：让 DeerFlow 发现工具

接入配置后，启动 DeerFlow，检查工具列表是否出现。

这一步只验证“能不能发现工具”，不要马上跑复杂任务。

如果工具不可见，优先检查配置文件路径、server 命令、URL、token、传输方式和 DeerFlow 日志。

第三步：调用一个只读工具

选择最简单、风险最低的工具。

例如查询版本、列出资源、搜索一个公开条目、读取一个测试页面。

确认输入参数和返回结果都正常。

第四步：让 Agent 在任务中使用

工具可用后，再给 Agent 一个明确任务：

请使用已配置的 MCP 工具读取测试资源，并总结返回内容。不要调用写入类工具。

观察 Agent 是否选择了正确工具，参数是否正确，返回内容是否被正确理解，是否误调用了高风险工具。

写入类工具要单独控制

如果 MCP server 暴露了写入能力，比如创建 issue、更新数据库、发送消息，必须单独设计权限。

建议只读和写入工具分开，写入操作需要用户确认，删除、发布、部署类工具默认关闭，记录调用日志，用测试环境先跑。

常见失败点

  • Server 起不来：先看依赖、启动命令和环境变量。
  • DeerFlow 连接不上：检查 URL、端口、传输方式、网络、日志。
  • 工具列表为空：可能是配置没加载、server 没返回 schema、权限不足。
  • Agent 不会用：可能是工具命名不清楚、描述太弱、参数 schema 太复杂。
  • 返回太大：需要分页、摘要或限制字段。

复盘

MCP 接入的关键不是“配置写上去”，而是分层验证。

server 可用、工具可见、只读调用成功、真实任务使用成功，这四步都通过，才算接入完成。
